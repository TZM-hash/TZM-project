# 阶段 7：PWA 有限离线与整体图形化设计

版本：v1.0  
日期：2026-07-16  
状态：已确认，直接执行

## 1. 目标与边界

阶段 7 交付两项能力：

1. 现场人员可离线保存阶段成果草稿、工程量、备注和照片，恢复网络后自动或手动同步。
2. 首页从阶段 0 占位页升级为真实经营驾驶舱，在电脑和手机上展示项目、财务、工资、风险及同步状态。

明确不进入离线范围的数据：合同金额、最终结算、收付款、发票、工资结算和账户流水。这些数据离线时只显示不可操作提示，不缓存业务明细。

浏览器支持最新版及前两个主要版本的 Windows Edge/Chrome、Android Chrome 和 iPhone Safari。旧浏览器保持在线页面可用，但不承诺离线草稿和照片同步。

## 2. 离线架构

### 2.1 客户端存储

浏览器使用版本化 IndexedDB 数据库，分为：

- `drafts`：阶段成果草稿、客户端草稿 ID、服务器 ID、服务器版本、项目、合同、类型、日期、说明和工程量行。
- `photos`：客户端照片 ID、所属草稿、压缩后 Blob、文件名、类型、大小和同步状态。
- `queue`：待同步操作、重试次数、最后错误、下次重试时间和状态。
- `metadata`：数据库版本、当前用户 ID、允许离线使用的项目/合同/清单选项和最后更新时间。

每份草稿最多 20 张离线照片。照片保存前在客户端压缩到最长边约 1920 像素、单张目标不超过约 3 MB；联网普通上传不强制压缩原图。使用 `navigator.storage.estimate()` 监测空间，接近限额时停止新增并给出中文提示。

### 2.2 用户隔离与清理

本地草稿必须带当前 Identity 用户 ID，页面只读取本用户分区。检测到当前登录用户与本地元数据不一致时，不显示另一用户数据并要求先清理。离线页提供“清除此设备离线数据”操作。

浏览器配置文件仍是本地数据的安全边界；共享电脑或手机应使用独立系统账户/浏览器配置并在交接设备前清理离线数据。

### 2.3 同步协议

服务端新增离线同步记录，使用 `(用户 ID, 客户端草稿 ID)` 唯一约束实现幂等。同步请求包含服务器阶段成果 ID 和上次服务器 `ConcurrencyStamp`：

- 首次同步创建正式的服务器端草稿并返回服务器 ID 与新版本。
- 相同客户端草稿重复提交只更新同一服务器草稿，不重复创建。
- 服务器版本与客户端基准版本一致时，替换草稿字段和工程量行并生成新版本。
- 服务器版本已变化时返回冲突和服务器快照，不自动覆盖。
- 已转为正式记录或作废的阶段成果禁止继续由离线草稿覆盖。

照片单独上传，使用 `(同步记录, 客户端照片 ID)` 唯一约束保证重试不重复。草稿先同步成功，再逐张同步照片；单张失败不影响其他照片。

### 2.4 失败和冲突

客户端队列采用递增退避，网络恢复、页面重新打开或用户点击“立即同步”时继续。`navigator.onLine` 只作为提示，实际请求结果才决定状态。

- 网络失败：保留本地草稿和照片，显示待同步/重试状态。
- 业务校验失败：停止该操作自动重试，显示具体字段原因。
- 并发冲突：并排给出本地摘要和服务器摘要，用户选择保留服务器版本或基于服务器版本重新提交本地内容。
- 服务端冲突、校验或照片失败写入同步记录，提醒中心生成“离线同步失败”提醒；纯断网状态先在本地首页显示，联网后再上报。

## 3. 服务端组件

- `IOfflineStageResultService`：项目选项、草稿同步、照片同步、失败上报和冲突解决。
- `OfflineDraftSync`：客户端草稿映射、服务器版本、同步状态、错误和时间。
- `OfflineAttachmentSync`：客户端照片映射和附件映射。
- Razor Pages JSON handlers：仅向系统管理员、应用管理员、项目负责人和现场人员开放。
- 现有 `StageResult`、`StageResultLine`、`Attachment`、`IFileStore` 和 `ConcurrencyStamp` 继续作为权威业务记录，不复制第二套阶段成果模型。

项目选项只返回当前用户可参与的有效项目；系统/应用管理员可返回全部有效项目。离线提交仍在服务端重新校验项目、合同、清单项和用户权限。

## 4. 首页经营驾驶舱

### 4.1 分析任务与展示族

主要分析任务是经营监控和风险定位。首页采用：

- KPI 指标卡：项目数、当前工程金额、未收款、未付款、未开发票、未发工资和未处理提醒。
- 项目阶段分布：按阶段显示项目数量的水平条形分布。
- 经营金额对比：应收/已收、应付/已付、应开票/已开票的成组水平进度条，关键数值始终可见。
- 风险列表：按严重度和时间排序，直接进入对应业务页面。
- 离线同步卡：由客户端补充本设备待同步、失败和冲突数量。

首页不使用 3D、动画背景、饼图或只靠悬停才能读数的图表。

### 4.2 渲染技术设计

- 同页实例：4～8 个指标卡、1 个项目阶段图、1 个经营金额对比图、1 个风险列表。
- 数据规模：项目阶段不超过约 12 个类别，金额对比固定 3 组，使用服务器聚合结果。
- 更新频率：每次页面加载查询，不做实时推送和持续动画。
- 渲染器：Razor 服务器渲染 + 语义化 HTML/CSS；条形图使用 CSS 宽度和可访问文本，不增加图表包。
- 降级：CSS 不可用时仍能按文本和表格读出全部数值；零值和无数据状态有明确说明。
- 状态：当前阶段不把首页筛选写入 URL，不保存个人仪表盘布局；数据截止时间显示在页面。
- 性能：不新增 JavaScript 图表依赖，移动端只重排布局，不重复请求同一汇总。

### 4.3 移动端与无障碍

桌面端先显示指标卡和两列分析区，手机纵向按“风险摘要 → 关键指标 → 金额对比 → 项目阶段 → 明细入口”排列。关键数值无需悬停；颜色同时配合文字和数值，不单独承担状态含义。触控按钮目标不小于约 44 CSS 像素，并支持键盘焦点和减少动态效果偏好。

匿名访问首页仍返回 200，但只显示登录入口和系统状态，不泄露业务数据。登录用户按现有角色和数据范围获取统计。

## 5. Service Worker

Service Worker 升级缓存版本并明确分流：

- 静态 CSS、JS、Manifest 和离线页面外壳使用缓存优先。
- 普通页面导航使用网络优先，只对离线阶段成果外壳提供缓存回退。
- `/api/`、财务、工资、合同、结算、导入导出、备份和提醒响应不写入 Cache Storage。
- 激活时删除旧版本缓存；部署新版本后提示用户刷新。
- 不拦截 POST/PUT/DELETE，不伪造业务写入成功；写入队列由 IndexedDB 同步模块负责。

## 6. 测试与验收

- 领域/应用测试：首次同步、幂等重试、版本更新、冲突、正式记录保护、照片幂等和失败状态。
- 数据模型测试：唯一索引、外键、长度和删除行为。
- 首页服务测试：空数据、项目阶段、经营金额、工资和提醒聚合。
- Web 授权测试：现场人员可进入离线页；未授权角色和匿名用户被拒绝；匿名首页不泄露指标。
- 静态资源测试：IndexedDB 存储结构、照片限制/压缩入口、Service Worker 敏感路由排除规则存在。
- 浏览器验收：桌面与手机视口、离线保存、刷新后保留、断网重试、冲突提示、单张照片失败重试、清理本机数据和重新联网同步。
- 阶段验收：生成并应用迁移，运行完整质量门禁，真实访问健康端点、首页、离线页和同步端点。
