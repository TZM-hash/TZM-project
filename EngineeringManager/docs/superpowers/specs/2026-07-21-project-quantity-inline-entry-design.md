# 项目工程量明细内联新增设计

## 背景

当前项目详情页的“工程量明细”标签页可以查看和快捷修改已有清单行，但新增工程量和附件需要跳转到 `Projects/Contracts/Edit` 独立页面。用户希望把新增操作集中到项目详情页，并让每条工程量直接拥有一个附件上传入口，从而减少页面跳转。

## 目标

1. 在项目详情页的工程量标签页中，以与“新增发票记录”相同的折叠交互提供“新增工程量明细”。
2. 删除工程量独立详细编辑页面及其所有入口；旧路由不再提供页面。
3. 在工程量表格的指定位置增加每行一个上传按钮，每条明细只允许一个有效附件；再次上传替换旧附件。
4. 将工程量表格前部列压缩并左靠，把最右侧“备注”列设为最大剩余宽度且允许换行。

## 非目标

- 不改变项目总览、合同金额计算、财务台账或工程量业务规则。
- 不把单附件限制扩展到收款、发票、付款、施工等其他业务记录。
- 不新增异步上传或前端单页应用框架；继续使用 Razor Pages 表单提交。

## 现状与约束

- 项目详情页位于 `src/EngineeringManager.Web/Pages/Projects/Details.cshtml` 及对应后台模型。
- 工程量创建目前位于 `src/EngineeringManager.Web/Pages/Projects/Contracts/Edit.cshtml`。
- `IProjectRecordAttachmentService` 已提供按业务明细列出、上传、下载和软删除附件的能力。
- 现有权限角色为系统管理员、应用管理员和项目经理；附件仍受现有项目归属校验和 20MB 文件限制保护。
- 当前工作区可能包含用户已完成的未提交页面优化，实施时必须保留与本功能无关的修改。

## 方案

### 页面与交互

工程量标签页保留已有“快捷编辑”入口，在其下增加 `details.inline-create-details` 折叠区域。折叠区域字段为：

- 所属合同
- 口径（暂估、结算）
- 新增附件和附件说明
- 编号、名称、单位
- 工程量、单价、小计预览
- 备注
- 是否需要开票

小计由工程量和单价在浏览器中实时预览，后端仍以提交值执行既有金额计算。保存成功后重定向回当前项目的工程量标签页，并使用新增行锚点定位。

工程量表格列顺序为：编号、名称、单位、工程量、单价、小计、口径、是否开票、上传、附件、备注。上传列位于“是否开票”和“附件”之间，对应用户标注位置；每行提供独立的文件选择和上传提交。附件列显示当前有效附件的预览链接、删除按钮或空状态。

表格使用固定或受控的紧凑列宽，前部字段整体左靠；备注列通过剩余空间占满可用宽度并设置自然换行。桌面端优先保证备注阅读空间，小屏幕保留横向滚动以避免控件重叠或不可操作。

### 页面删除与入口清理

删除 `Projects/Contracts/Edit.cshtml`、对应 `.cshtml.cs` 及仅服务于该页面的页面测试。移除项目详情页的“进入详细编辑”按钮、工程量名称到该页面的链接、附件到该页面的链接和相关锚点依赖。原路由不再映射到页面，访问时返回 404。项目详情页继续提供已有行快捷编辑和新的折叠新增。

### 后端数据流

1. 项目详情页新增工程量 POST handler 校验项目和合同，再调用现有 `IProjectService.AddLineItemAsync`。
2. 新增请求若包含附件，调用附件服务保存到新工程量行。
3. 详情页加载项目工作区后，为每条工程量构建附件字典，供表格展示。
4. 行级附件上传 handler 校验当前用户、项目和工程量归属，调用“单附件替换”能力。
5. 替换逻辑先完成新文件保存，再在同一业务操作中将原有效附件标记为删除并保存新记录；新文件保存失败时旧附件保持有效。
6. 行级删除和预览继续复用现有下载、软删除接口。

为了避免影响其他业务记录，单附件替换只对工程量明细开放，不改变通用 `UploadAsync` 对其他记录允许多附件的行为。替换操作应通过附件服务提供明确的工程量专用方法或等价的事务边界实现，而不是在 Razor Page 中拼接删除和上传步骤。

### 错误处理与权限

- 没有项目、合同或工程量归属时拒绝操作并返回既有错误提示。
- 文件为空、超过 20MB、文件名无效或无管理权限时拒绝上传。
- 新增表单校验失败时返回当前页面，保留用户输入并保持新增折叠区打开。
- 附件替换失败时不删除旧附件，不影响工程量记录。
- 继续使用现有授权角色和防伪令牌，不新增权限键。

## 验证策略

### 单元/应用服务测试

- 工程量附件首次上传成功。
- 已有工程量附件再次上传后只有一个有效附件，旧附件标记删除。
- 新文件校验或文件存储失败时旧附件仍有效。
- 其他类型记录仍保留原有多附件行为。

### Web 页面测试

- 项目经理可以看到折叠新增区域、上传列和附件控件。
- 工程量标签页不包含“进入详细编辑”、独立页面链接或旧页面专用锚点。
- 页面源码包含备注最大宽度布局选择器和上传列结构。
- 无权限角色不能打开创建或上传入口。
- 旧独立页面文件不存在，旧路由返回 404。

### 构建与页面验证

- 运行目标 Web 测试项目及完整 `dotnet test`。
- 运行 `dotnet build` 验证 Razor 编译和依赖清理。
- 启动本地 Web 应用，用浏览器检查桌面和窄屏工程量标签页：折叠新增、保存定位、行级上传/替换/删除、备注列宽度及无横向重叠。

## 验收标准

- 用户在项目详情页展开“新增工程量明细”即可完成新增，无需访问其他页面。
- 图1独立详细编辑页面和图2对应入口均已删除。
- 每条工程量在标注位置有上传按钮，且任何时刻最多一个有效附件。
- 再次上传替换旧附件，失败不丢失旧附件。
- 工程量表格前部紧凑左靠，备注列在桌面端获得最大显示空间，内容不遮挡、不溢出。
- 自动化测试、构建和浏览器验证均通过。
