# 统一财务账本、年度账与阶段对账点设计

**日期：** 2026-07-18
**状态：** 已被 `2026-07-19-central-ledger-multi-entry-design.md` 取代，仅保留为历史设计记录

> 2026-07-19 用户重新确认需求：保留项目、班组、合作商多入口录入，新增外部/内部中央账本，并调整超前发票收付、独立财务年度、可修正对账和物理删除审计规则。后续实施以新规格为准。

## 1. 目标

将项目、自有公司、班组、合作商中的结算、发票、收款和付款统一到同一套中央财务账本。每条业务单据只保存一次，各模块根据参与方、项目和分摊关系展示同一条记录，并允许从任一有权限的关联页面修改。

系统需要同时支持：

- 每家自有公司的发票、收款、付款和结算管理。
- 项目、班组、合作商与自有公司之间的双向查看和编辑。
- 销项发票、进项发票、内部发票、外部收付款和内部转账。
- 一张发票或一笔资金跨多个项目、合同和结算明细分摊。
- 暂不分摊或部分分摊，剩余金额后续处理。
- 累计总账、年度账、跨年余额和阶段对账结转。
- 集团简单合计和抵销内部往来后的经营口径。
- 已确认单据留痕修改，以及对账点确认后的历史冻结。

## 2. 范围

本设计覆盖：

- 中央结算单、发票、资金单和分摊关系。
- 自有公司、项目、班组、合作商和全局财务核对页面。
- 工资代发对班组应付的冲减。
- 年度统计、跨年结转和阶段对账点。
- 内部往来双视角和集团抵销。
- 修改审计、异常提示和权限边界。

本设计不引入完整复式会计总账、会计科目、凭证过账或法定财务报表，也不要求把付款精确核销到某一张具体发票。

## 3. 核心原则

### 3.1 单一数据源

来源模块不保存财务副本。项目、自有公司、班组、合作商和财务中心全部读取并修改中央记录。来源信息只用于预填、权限校验、审计和回跳。

### 3.2 独立业务单据

结算、发票和资金是相互独立的业务事实：

- 结算表示双方确认的债权或债务。
- 发票表示税务票面事实。
- 资金表示实际收款、付款或内部转账。
- 分摊表示这些事实与项目、合同和结算明细之间的金额关系。

### 3.3 分摊而非复制

一张发票或一笔资金可以分摊到多条结算明细。分摊金额可以跨项目，分摊总额不能超过单据有效金额。未分摊余额单独保留。

### 3.4 统计按最小结算明细计算

开票与收付款的交叉统计先在每条结算明细内计算，再向项目、合同、往来单位和公司汇总，禁止直接用无关业务的总额互相抵消。

## 4. 参与方模型

业务参与方支持：

- 自有公司 `LegalEntity`。
- 外部单位 `BusinessPartner`；班组是合作单位的一种业务角色。

单据保存业务意义明确的双方，而不是只保存“当前查看公司”：

- 结算保存债权方和债务方。
- 发票保存销售方和购买方。
- 资金保存付款方和收款方。

当双方都是自有公司时，单据自动标记为内部往来。同一记录在一方显示为销项、应收或收款，在另一方显示为进项、应付或付款。

## 5. 中央单据

### 5.1 结算单

结算单头至少包含：

- 单据编号、结算日期和状态。
- 债权方、债务方。
- 创建来源、备注、附件、版本和审计信息。

一张结算单可以包含多条结算明细。每条明细至少包含：

- 项目和可选合同或合同清单。
- 结算金额。
- 应开票金额。
- 实际应收或应付金额。
- 业务分类和说明。

项目可以为空，以支持公司级零散业务。

### 5.2 扣款与抵扣

扣款或抵扣关联到具体结算明细，保存金额、类型、日期、说明和状态。每条记录提供 `同时减少应开票金额` 选项，默认不勾选。

计算口径为：

```text
实际应收/应付金额 = 结算金额 - 全部有效扣款或抵扣
应开票金额 = 结算金额 - 勾选“同时减少应开票金额”的有效扣款或抵扣
```

两个结果均不得小于零。修改已确认扣款的金额或该选项时，必须填写原因并记录前后快照。

### 5.3 发票

发票至少保存：

- 销售方、购买方和当前自有公司视角。
- 发票号码、日期、方向和状态。
- 普票或专票、税率、不含税金额、税额和价税合计。
- 项目税金配置引用；公司级零散发票允许不引用项目税金配置，但仍需使用有效的结构化票种和税率。
- 创建来源、附件、版本和审计信息。

发票方向从销售方、购买方与当前公司之间的关系推导；内部发票只保存一次。

发票分摊明细保存项目、可选结算明细和分摊金额。没有结算单时可以直接关联项目；项目也可以为空。作废发票不进入正式统计，红冲通过关联的更正记录冲减有效金额。

发票号码按销售方和发票号码执行唯一性或重复预警，避免把不同销售方的合法同号票误判为同一张发票。

### 5.4 资金单

资金单统一表示收款、付款和内部转账，至少保存：

- 付款方、收款方。
- 付款账户和可选收款账户。
- 日期、金额、支付方式和状态。
- 创建来源、附件、版本和审计信息。

资金分摊明细保存项目、可选结算明细和分摊金额。预收、预付或暂时无法确认归属的金额可以不分摊或只分摊一部分。

内部转账使用一张资金单和双方账户，只产生一次业务动作；账户流水是资金单的受控投影，不是第二份可独立编辑的数据。

退款和冲销使用关联的反向记录冲减有效收付金额，不物理删除原记录。

### 5.5 工资代发

民工工资代发批次可以关联班组结算明细。关联后作为该班组的代付工程款冲减实际应付，但在统计中与直接支付给班组、其他抵扣分开显示。

未关联结算的工资代发进入待分摊，不直接冲减某个班组或项目的未付款金额。

## 6. 分摊规则

- 一张发票可以有多条发票分摊明细。
- 一笔资金可以有多条资金分摊明细。
- 分摊可以跨项目、合同和结算单。
- 分摊金额必须大于零。
- 同一单据的有效分摊总额不得超过单据有效金额。
- 待分摊金额等于单据有效金额减去有效分摊总额。
- 关联结算明细时，项目、参与方和业务方向必须一致。
- 修改跨项目单据时，页面必须明确提示会影响其他项目。
- 本系统不要求资金精确核销到具体发票；开票与收付交叉状态按结算明细汇总推算。

## 7. 统计口径

每条结算明细定义：

- `S`：结算金额。
- `B`：应开票金额。
- `N`：实际应收或应付金额。
- `I`：分摊到该明细的有效发票金额。
- `P`：分摊到该明细的有效收款或付款金额。
- `O`：扣款、抵扣及其他非现金结清金额。

基础指标：

```text
已开票 = I
未开票 = max(B - I, 0)
超额开票 = max(I - B, 0)
已收/已付 = P
未收/未付 = max(N - P, 0)
超收/超付 = max(P - N, 0)
非现金抵扣 = O
```

交叉统计在实际应收或应付基数 `N` 内计算。先令：

```text
有效开票覆盖 = min(I, N)
有效资金覆盖 = min(P, N)
```

再计算：

```text
已开票已收/已付 = min(有效开票覆盖, 有效资金覆盖)
已开票未收/未付 = max(有效开票覆盖 - 有效资金覆盖, 0)
未开票已收/已付 = max(有效资金覆盖 - 有效开票覆盖, 0)
未开票未收/未付 = max(N - max(有效开票覆盖, 有效资金覆盖), 0)
```

发票金额超过实际应收或应付基数、但未超过应开票基数的部分，单独显示为“已开票但无需现金收付”；超过应开票基数的部分显示为超额开票。

所有指标先逐条结算明细计算再汇总。草稿默认不进入正式统计，已确认记录进入统计，已作废记录排除但保留历史。

## 8. 统计维度

所有总账和明细查询统一支持：

- 自有公司和集团。
- 项目、合同和合同清单。
- 班组、合作商或客户。
- 结算单和结算明细。
- 日期区间和业务年度。
- 发票方向、票种、税率和状态。
- 内部或外部业务。
- 已分摊、部分分摊和未分摊。

公司统计按单据参与方统计全部业务；项目统计只使用分摊给该项目的金额；往来单位统计按参与方统计，并单独展示尚未归属项目的金额。

集团提供：

- 公司简单合计。
- 内部往来金额。
- 抵销内部往来后的集团经营数据。

抵销只影响集团统计展示，不修改各公司原始账目。

## 9. 累计总账与年度账

自有公司、班组和合作商均提供：

- 累计总账：从最早记录到当前日期。
- 年度账：期初结转、本年发生和期末余额。

年度账至少分别展示：

- 上年未收或未付结转。
- 上年未开票结转。
- 上年已开票未收或未付结转。
- 上年超收或超付结转。
- 本年新增结算、开票、收付和抵扣。
- 本年期末余额。

年度账复用现有不重叠的业务年度定义。年度结束可以自动生成阶段对账草稿，但必须人工确认。

## 10. 阶段对账点

### 10.1 对账范围

用户可以按自有公司、班组或合作商选择截止日期生成阶段对账单。对账单按公司、往来单位、项目和合同保存结构化明细，不只保存总数。

### 10.2 快照内容

对账快照至少包含：

- 结算金额、应开票金额和实际应收或应付。
- 已开票、未开票和超额开票。
- 已收或已付、未收或未付、超收或超付。
- 四类开票与收付交叉指标。
- 扣款、抵扣和待分摊金额。
- 对账编号、期间、截止日期、备注、附件、确认人和确认时间。

### 10.3 冻结与结转

用户选择冻结式对账点：

- 对账点确认后，截止日期及以前的业务记录只读。
- 未完成金额转换为下一期间的期初结转明细。
- 后续发票、收付款可以直接分摊到期初结转，无需打开历史结算单。
- 历史错误在对账点之后创建更正单，并引用原对账点。
- 日常列表默认只显示期初结转和对账点之后的数据。
- 历史依据仍可从对账单查看。

系统优先建议处理最早的期初结转，再处理本期新增结算，但用户可以调整分配。

同一账务范围的对账点必须按日期递增且不得重叠。只有最新对账点、且不存在后续对账点时，管理员才可以填写原因后撤销确认。

## 11. 页面设计

### 11.1 自有公司

每家公司提供：

- 经营总览。
- 发票管理，分段切换销项、进项和内部发票。
- 收款管理。
- 付款管理。
- 结算管理。
- 待分摊。
- 内部往来。
- 累计总账、年度账和阶段对账单。

### 11.2 项目

项目页显示分摊给当前项目的金额，同时显示单据总额。新增时预填项目；编辑跨项目单据时提示影响范围并进入统一编辑器。

### 11.3 班组

班组详情显示结算、应开票、开票、直接付款、工资代发、其他抵扣、剩余应付及交叉统计，并提供累计、年度和阶段对账视图。

### 11.4 合作商

合作商详情复用同一结算、发票和资金数据。具有客户角色时可以切换查看销项发票与收款；具有供应商或班组角色时查看进项发票与付款。

### 11.5 全局财务核对中心

集中列出：

- 未分摊或部分分摊单据。
- 超额开票、超收和超付。
- 发票号码疑似重复。
- 已确认后修改的记录。
- 工资代发尚未关联班组结算。
- 作废、红冲、退款和冲销记录。
- 待确认的年度或阶段对账草稿。

所有入口使用同一中央单据详情和编辑服务，来源页面不实现独立保存逻辑。

## 12. 修改、并发与审计

- 草稿可以正常修改。
- 已确认的结算、发票和收付款仍可修改，但必须填写原因。
- 每次修改保存操作人、时间、来源页面、修改前后快照和并发版本。
- 对账点确认后，截止日期以前的数据只读；错误通过后续更正单处理。
- 已作废记录不参与统计且不能恢复为普通可编辑状态。
- 被其他用户修改的旧版本保存请求必须拒绝并提示刷新。
- 已被分摊或引用的记录不得物理删除。

## 13. 权限

- 查看权限同时受角色、公司范围和项目范围约束。
- 新增和编辑单据必须具有相关公司的财务管理权限。
- 修改跨项目或跨公司的分摊时，必须对所有受影响范围具有权限。
- 只读用户只能查看和导出授权范围数据。
- 阶段对账确认、撤销和历史更正使用独立的高权限操作。
- 集团抵销口径只向具有全部相关公司访问权限的用户展示。

## 14. 迁移方向

现有 `InvoiceEntry`、`ReceivableEntry`、`PayableEntry`、`CollectionEntry`、`PaymentEntry` 和关联表需要迁移到中央单据及分摊结构。迁移遵循：

- 保留原始 ID 或建立稳定的旧 ID 映射。
- 现有项目记录生成对应项目分摊。
- 现有发票税率、票种和项目税金配置保留。
- 现有收款和付款生成资金单及项目分摊。
- 现有工资代发按已有班组、项目和应付关联生成代付分摊；无法确认的进入待分摊。
- 迁移前生成冲突报告，不静默猜测缺失关系。
- 迁移后对项目、公司和全局统计执行金额对账。

数据库迁移必须先在测试库执行，并保留可验证的迁移前后汇总结果。

## 15. 实施分期

统一财务改造按依赖顺序拆成四个可独立验收的实施阶段：

1. 中央账本基础：参与方、结算单、扣款、发票、资金单、分摊、审计和迁移前对账工具。
2. 业务入口统一：迁移现有财务记录，改造公司、项目、班组、合作商和全局核对页面，停止模块内财务副本写入。
3. 汇总与内部往来：交叉统计、待分摊、工资代发、累计总账、年度账和集团内部抵销。
4. 阶段对账：结构化快照、冻结、期初结转、后续更正、撤销限制和年度对账草稿。

后一阶段不得在前一阶段的迁移对账和回归测试通过前投入使用。临时人员并入员工管理使用独立规格和实施计划，可以先行实施，但其历史工资迁移必须为第三阶段的工资代发接入保留稳定员工引用。

## 16. 测试与验收

至少覆盖：

- 单据从任一关联页面修改后，其他页面读取同一记录。
- 发票和资金跨项目、跨结算分摊。
- 部分分摊、待分摊和后续继续分摊。
- 进项、销项和内部发票双视角。
- 内部资金转账双方视角及集团抵销。
- 扣款是否影响应开票金额的两种情况。
- 四类开票与收付交叉统计、超额开票、超收和超付。
- 工资代发关联班组结算和未关联待分摊。
- 累计总账、年度期初结转和跨年余额。
- 阶段对账快照、冻结、期初结转和后续更正。
- 并发冲突、审计快照和权限范围。
- 历史迁移前后金额总计一致。

验收时必须分别核对公司、项目、班组、合作商和集团抵销口径，且任何正式统计都能下钻到原始单据、分摊明细或对账结转来源。
