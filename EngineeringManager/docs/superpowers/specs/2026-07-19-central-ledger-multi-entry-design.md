# 中央财务账本与多入口统一财务设计

**日期：** 2026-07-19
**状态：** 已完成用户逐项确认，并于 2026-07-20 完成实现与本地验收
**取代规格：** `2026-07-18-unified-finance-reconciliation-design.md`

## 1. 目标

在保留项目管理、施工班组管理和合作商管理现有财务录入入口的基础上，新增统一的中央账本。所有入口共用同一份底层财务记录，不在各模块复制应收、应付、发票或收付款数据。

中央账本用于：

- 汇总全部授权范围内的应收、应付、收款、付款、销项发票、进项发票、扣款和调整。
- 从中央账本直接新增、修改和删除财务记录。
- 按自有公司、项目、合同、施工班组、合作商、财务年度、日期和业务状态筛选。
- 独立统计未收、未付、未开票、开票已收、开票已付、开票未收、开票未付、超前发票收款、超前发票付款、超结算收付和超额开票。
- 支持暂估结算、最终结算差额、跨项目分摊、待分摊、独立财务年度、跨年动态结转和可修正对账快照。
- 保留每次新增、修改、删除、作废、冲销、分摊和对账操作的完整审计记录。

本设计不建设复式会计、会计科目、会计凭证或法定财务报表，也不增加审批流程。业务确认后直接形成正式财务记录，审批可在未来独立扩展。

## 2. 核心原则

### 2.1 同一记录，多入口使用

项目、班组、合作商和中央账本只是不同业务入口。任一入口新增或修改记录后，其他关联入口读取同一条数据并立即反映变化。

来源页面只负责提供业务上下文、默认值、权限范围和回跳地址，不保存中央记录的副本。

### 2.2 项目统计与财务年度相互独立

项目页面始终按项目全生命周期统计，不受财务业务年度切割。中央账本使用独立财务业务年度对同一批记录进行年度归属、年度汇总和跨年结转展示。

员工管理继续使用现有员工业务年度。员工业务年度与财务业务年度互不共用、互不限制，可以采用完全不同的起止日期。

### 2.3 原始事实与自动指标分离

用户维护的原始事实包括结算、扣款、发票、收款、付款、退款、冲销和调整。未收、未付、未开票、组合状态和异常金额由系统按明细实时计算，不允许用户直接覆盖计算结果。

### 2.4 先按明细计算，再汇总

所有财务指标先在单条结算明细内计算，再向项目、合同、班组、合作商、自有公司和中央账本汇总，禁止用无关业务总额互相抵消。

## 3. 导航与页面结构

左侧导航新增可折叠的二级菜单：

```text
中央账本
├─ 外部账本
└─ 内部账本
```

菜单记住当前用户上次的展开或折叠状态。

外部账本与内部账本均采用“二级菜单进入页面、页面内标签切换业务类别”的结构，不增加三级导航。

外部账本页面标签：

```text
总览
应收
应付
收款
付款
销项发票
进项发票
扣款
待分摊
异常
年度账
对账记录
修改日志
```

内部账本只显示自有公司之间的内部应收、应付、发票和资金转账，不混入外部账本指标。外部账本和内部账本共用同一套财务业务年度。

## 4. 业务入口规则

### 4.1 项目管理

- 工程量输入和确认固定形成项目应收，不允许改为应付。
- 未最终结算时可以输入暂估结算金额，并按暂估金额计算当前应收和开票进度。
- 项目收款明细只记录实际收款。
- 项目付款明细只记录实际付款，收款方可以是施工班组或合作商。
- 项目开票明细只录入销项发票。

### 4.2 施工班组和合作商管理

- 可以单独新增应付款，并可关联项目、合同或不关联项目。
- 可以录入进项发票。
- 可以查看和录入相关付款、扣款及分摊。
- 班组是合作单位的一种业务角色，但界面和筛选中继续使用明确的“施工班组”名称。

### 4.3 中央账本

- 外部账本可以新增应收、应付、收款、付款、销项发票、进项发票、扣款和调整。
- 新增时选择自有公司、往来单位和可选项目/合同；选择的上下文决定记录出现在哪些业务页面。
- 内部账本单独新增和维护自有公司之间的内部应收、应付、发票和转账。

## 5. 统一数据模型

### 5.1 结算与应收应付

统一结算记录至少包含：

- 外部账或内部账范围。
- 应收或应付方向。
- 自有公司和往来方；内部账的双方均为自有公司。
- 可选项目、合同、合同清单、施工班组或合作商。
- 来源模块、来源记录 ID 和回跳地址。
- 暂估或最终结算状态、结算日期和业务日期。
- 原始结算金额、原始应开票金额、备注和并发版本。

项目工程量确认生成应收结算。班组、合作商或中央账本可以创建应付结算。业务确认后直接形成正式记录，不经过审批或待确认状态。

### 5.2 暂估与最终结算

暂估结算可以正常关联发票、收付款和扣款。

最终结算确认时不覆盖暂估历史，而是保存最终金额并生成可追溯差额：

```text
结算调整额 = 最终结算金额 - 暂估结算金额
```

正差额增加当前应收或应付，负差额减少当前应收或应付。调整保留来源、原因、操作人、时间和前后值。

### 5.3 扣款

每条扣款始终减少实际应收或应付，并提供“同时扣减应开票金额”选项：

- 勾选：同时减少实际应收/应付和应开票金额。
- 不勾选：只减少实际应收/应付，应开票金额保持不变。

班组付款和合作商付款入口可以同时录入实际付款及多条扣款。扣款属于非现金结清，不计入“已付款”。

### 5.4 发票

- 项目管理新增销项发票。
- 施工班组和合作商管理新增进项发票。
- 外部中央账本可以新增销项或进项发票。
- 内部账本维护自有公司之间的内部发票。

一张发票可以通过分摊明细关联多个项目、合同和结算记录。发票作废或红冲保留历史并按有效金额重新计算。

### 5.5 资金记录

资金记录区分收款、付款和内部转账，至少保存双方、付款账户、可选收款账户、日期、金额、支付方式、备注和来源。

一笔收款或付款可以跨多个项目、合同和结算记录分摊。没有可匹配结算或存在剩余金额时，余额保留为待分摊。

退款和付款冲销使用关联的反向记录计算有效金额，也允许按本设计的物理删除规则删除错误记录。

### 5.6 分摊

发票和资金分别使用独立分摊明细。每条分摊保存：

- 目标结算记录。
- 项目和可选合同。
- 班组或合作商。
- 分摊金额、顺序、来源和并发版本。

用户可以手工分摊。未手工指定时，系统在同一自有公司、同一往来单位范围内，按最早未结清或未开票记录优先自动分摊。用户可以在之后重新调整分摊。

## 6. 金额与统计口径

对每条结算明细定义：

- `G`：原始暂估或最终结算金额。
- `D`：全部有效扣款。
- `DI`：勾选“同时扣减应开票金额”的有效扣款。
- `B0`：原始应开票金额，默认等于结算金额但允许单独填写。
- `I`：分摊到该结算明细的有效发票金额。
- `C`：分摊到该结算明细的有效实际收款或付款金额。

基础金额：

```text
实际应收/应付 N = max(G - D, 0)
当前应开票 B = max(B0 - DI, 0)

未收/未付 = max(N - C, 0)
超结算收款/付款 = max(C - N, 0)

未开票 = max(B - I, 0)
超额开票 = max(I - B, 0)
```

开票与收付组合指标：

```text
开票已收/已付 = min(I, C, N)
开票未收/未付 = max(min(I, N) - C, 0)
超前发票收款/付款 = max(C - I, 0)
未开票未收/未付 = max(N - max(min(I, N), min(C, N)), 0)
已开票但无需继续收付 = max(min(I, B) - N, 0)
```

“超前发票收款/付款”与“超结算收款/付款”是两个独立指标，可以同时出现：

- 超前发票收款/付款表示实际收付超过已开票金额。
- 超结算收款/付款表示实际收付超过暂估或最终结算金额，属于异常提示。

扣款单独显示为“扣款/非现金抵扣”，不进入实际已付款金额。

## 7. 财务业务年度

新增独立 `FinanceBusinessYear`：

- 年度名称、开始日期和结束日期由用户自定义。
- 财务年度之间不得重叠。
- 外部账本和内部账本共用财务年度。
- 财务年度只影响中央账本，不改变项目全生命周期统计。
- 员工业务年度继续独立运行。

跨年期初余额从上一财务年度期末结果动态计算。历史记录发生修正后，相关年度的期初、发生额和期末余额自动重新计算，不保存不可修改的固定年末余额。

年度账至少展示：

- 期初未收、未付、未开票、开票未收和开票未付。
- 本年新增结算、扣款、开票、收款和付款。
- 本年超前发票收付、超结算收付和超额开票。
- 期末未收、未付、未开票及待分摊余额。

## 8. 对账快照

用户可以按财务年度、任意截止日期、自有公司、往来单位或全账本生成年度或阶段对账快照。

对账快照不锁定历史数据。后续允许跨年度修正原始记录，但必须保留修改日志。

每份对账记录保存：

- 生成时的全部汇总指标。
- 按公司、项目、合同、班组、合作商和结算明细拆分的结构化快照。
- 当前重新计算值。
- 快照值与当前值的差异。
- 差异对应的修改、删除、冲销和重新分摊日志。

重新对账生成新版本，不覆盖历史快照。

## 9. 物理删除与审计

财务记录允许物理删除，但删除前必须写入不可修改的删除审计记录，至少包含：

- 删除人、删除时间、删除入口和必填删除原因。
- 被删除单据的全部字段和并发版本。
- 删除前的结算、开票、收付、扣款和统计金额。
- 关联公司、项目、合同、班组、合作商和分摊关系快照。
- 删除前后受影响的汇总金额。

删除规则：

- 删除结算或应收应付时，不级联删除发票和资金记录，只解除分摊；关联发票和资金转为待分摊。
- 删除发票、收款或付款时，只删除本单据及其分摊，不删除结算记录。
- 删除扣款后自动恢复实际应收/应付，并按扣款选项恢复应开票金额。
- 所有删除操作需要二次确认、必填原因和并发检查。

普通修改、新增、作废、红冲、冲销、分摊调整和对账操作继续写入现有审计体系，保存操作人、时间、入口、原因和前后 JSON。

## 10. 筛选、下钻与导出

中央账本统一支持：

- 财务业务年度和自定义日期区间。
- 自有公司。
- 项目、合同和合同清单。
- 施工班组、合作商和往来角色。
- 暂估或最终结算。
- 应收或应付方向。
- 已结清、部分结清或未结清。
- 已开票、部分开票或未开票。
- 已分摊、部分分摊或待分摊。
- 超前发票收付、超结算收付和超额开票。
- 单据编号、发票号码、项目名称、班组名称、合作商名称和备注全文关键词。

每条记录可以下钻查看来源、分摊、结算、扣款、发票、资金和修改历史，并可以跳回原始业务页面。

中央账本导出必须使用与页面相同的授权范围、筛选条件和统计口径。当前项目工作簿中的财务工作表需要改为读写新中央财务结构。

## 11. 权限与并发

- 页面和服务同时执行权限校验。
- 查看权限受自有公司、项目和角色范围约束。
- 新增或修改跨项目、跨公司分摊时，用户必须拥有所有受影响范围的管理权限。
- 只读用户只能查看和导出授权范围数据。
- 内部账本使用独立权限键，不因拥有外部账本权限而自动开放。
- 并发版本不一致时拒绝保存或删除，并提示用户刷新后重新操作。

## 12. 数据迁移

实施前必须备份 `EngineeringManager_Test`，所有迁移和金额核对只在测试库执行。生产数据库迁移需要未来再次单独确认。

迁移范围包括现有：

- `ReceivableEntry`、`PayableEntry`。
- `CollectionEntry`、`PaymentEntry`。
- `InvoiceEntry` 及发票关联。
- 退款、付款冲销、扣款、账户转账和账户流水。
- 工资班组分摊中可明确关联的财务数据。

迁移要求：

- 保留原始 ID 或建立稳定旧 ID 映射。
- 生成项目、公司、班组和合作商维度的迁移前汇总。
- 迁移后使用相同口径重新汇总并逐项核对。
- 无法确认关系的数据进入明确的待分摊或迁移异常清单，不静默猜测。
- 验收前保留旧表，但应用停止向旧表重复写入。
- 旧页面、中央账本和项目工作簿全部切换到新服务后再评估旧表清理；清理不属于本轮自动执行范围。

## 13. 实施分期

### 阶段一：中央账本基础

建立统一结算、扣款、发票、资金、分摊、删除审计和计算器，覆盖暂估、最终结算差额及核心统计公式。

### 阶段二：多入口接入与历史迁移

迁移现有财务数据，改造项目、班组、合作商和中央账本入口，使所有入口读写同一记录，并同步改造项目工作簿财务表。

### 阶段三：财务年度、内部账本与综合查询

实现独立财务年度、跨年动态结转、外部/内部账本页面、统一筛选、待分摊、异常和导出。

### 阶段四：对账、审计与完整验收

实现可修正对账快照、删除日志、差异追踪、权限并发、SQL Server 迁移对账、性能和浏览器验收。

后一阶段不得在前一阶段定向测试和金额核对通过前投入使用。

## 14. 测试与验收

至少覆盖：

- 项目工程量固定生成应收，暂估转最终结算生成正负差额。
- 原始应开票金额与结算金额可以不同。
- 扣款始终减少实际应收/应付，按选项决定是否减少应开票。
- 班组和合作商付款扣款不计入实际已付款。
- 发票和资金跨项目、合同及结算记录分摊。
- 最早未结清自动匹配、手工调整和待分摊。
- 未收、未付、未开票、开票已收/付、开票未收/付、超前发票收/付款。
- 超结算收付、超额开票和已开票但无需继续收付。
- 项目、班组、合作商和中央账本多入口读取同一记录。
- 外部账本和内部账本数据及权限隔离。
- 财务年度与员工年度隔离，项目全生命周期统计不受财务年度影响。
- 跨年动态结转和历史修正重新计算。
- 对账快照不锁定历史，能够展示快照、当前值、差异和修改来源。
- 物理删除保留完整日志，且不级联删除其他业务单据。
- 并发冲突、跨范围权限和审计前后快照。
- 迁移前后项目、公司、班组、合作商及全局金额一致。
- 中央账本筛选、导出和项目工作簿使用相同授权及统计口径。

最终验收必须运行定向领域/服务/迁移/页面测试、Release 全量测试、质量门禁、EF 模型变化检查、SQL Server 测试库迁移对账、桌面浏览器验证和约定的最终移动端验收。
