## 1. 文档目的

本文档将已确认的产品范围、业务规则、系统边界、数据模型、页面设计、计算口径、权限策略、导入导出、备份、离线能力和测试要求固化为实施依据。

本文档不是代码实现计划，也不授权立即修改或初始化项目。用户确认本文档后，才进入实施计划和项目初始化阶段。

## 2. 产品定位

系统面向本公司工程分包经营管理场景，服务于项目、合同、工程清单、阶段成果、甲方/总包、施工班组、材料供应商、零星供应商、内部员工、工资和经营台账管理。

系统的目标是把项目经营数据理清、打通并可追溯，而不是建设完整会计软件或对外供应商平台。

### 2.1 核心目标

- 以项目为最高管理单位，统一查看合同、工程量、阶段成果、收款、开票、付款和人员成本。
- 支持多家我方签约公司共同参与同一项目，并按公司分别核算后汇总到项目。
- 支持施工班组、材料供应商和零星供应商跨项目管理。
- 支持正式员工和劳务员工，以及阶段工资、报销、借支和其他员工往来。
- 支持暂估工程量/单价与最终结算工程量/单价并存，自动计算暂估金额、部分结算金额和结算金额。
- 提供可筛选、可配置字段、可保存设置的 Excel 汇总导出，直接服务领导台账和经营分析。
- 支持老系统数据平移、新系统 Excel 日常导入导出、数据库和附件备份迁移。
- 电脑端和手机端使用同一套响应式 Web/PWA，手机端只对阶段成果资料提供有限离线能力。

### 2.2 明确非目标

第一阶段不包含：

- 原生 Android/iOS App。
- 对外供应商或施工班组登录入口。
- 完整会计科目、会计凭证、总账、资产负债表、利润表和税务申报。
- 审批工作流和可视化流程设计器。
- 完整甘特图、复杂前置关系和任务调度。
- 完整股权持股、转让、稀释、回购或归属管理。
- 复杂通用字段映射平台。
- 多家互不相关企业的 SaaS 租户隔离。

## 3. 总体技术方案

### 3.1 方案选择

采用全新的模块化单体项目，选择性复用 `Reference` 的界面、组件和成熟基础能力，不直接复制旧项目作为新项目代码基础。

采用该方案的原因：

- 保留现有参考项目在权限、审计、Excel、附件、列表和 UI 方面的成熟经验。
- 避免旧项目的业务模型、繁体中文假设和历史模块约束新系统。
- 适合第一阶段快速交付，同时保留 API、审批、会计和原生客户端的升级空间。

### 3.2 运行形态

- 后端：ASP.NET Core on .NET 10 LTS。
- 数据库：SQL Server。
- Web：Razor Pages/服务端页面为主，配合页面级 JavaScript 和少量 API。
- 部署：Windows Server + IIS。
- 前端：简体中文、响应式布局、PWA 安装能力。
- 文件：开发阶段本地服务器目录；通过存储抽象接口预留云对象存储切换能力。
- 测试：服务层、数据库集成、页面权限、导入导出、离线同步和响应式 UI 测试。

### 3.3 分层边界

建议使用以下逻辑层：

1. **Presentation**：页面、API、PWA 同步端点、导出端点和权限展示。
2. **Application**：项目、合同、台账、工资、导入导出、备份等用例服务。
3. **Domain**：金额计算、合同分摊、暂估/结算、工资项目和状态规则。
4. **Infrastructure**：EF Core、SQL Server、文件存储、身份认证、后台任务和日志。
5. **Tests**：领域计算、服务、数据库、页面和浏览器交互测试。

业务页面不得绕过服务层直接编写复杂金额、权限或状态逻辑。

## 4. 模块边界

### 4.1 用户、组织与权限

- 用户账号、启用/停用、密码和登录安全。
- 系统级管理员、应用级管理员、财务人员、项目负责人、现场人员、查询人员等角色模板。
- 部门、分支机构和我方签约公司。
- 按签约公司、部门、项目、合作单位类型、数据类型设置数据范围。
- 单个用户权限覆盖。

### 4.2 项目与合同

- 项目主档和项目阶段。
- 甲方/总包合同、对外合同、补充协议和变更单。
- 工程清单项、暂估量价、结算量价。
- 多我方公司共同签约和合同分摊。
- 项目节点计划和项目归档。

### 4.3 阶段成果与完工资料

- 阶段成果记录。
- 清单项本期、累计和剩余工程量。
- 阶段资料、验收资料、照片和完工资料附件。

### 4.4 合作单位

统一合作单位档案，通过角色标签区分：

- 甲方/总包。
- 施工班组。
- 材料供应商。
- 零星供应商。

同一个单位可以有多个角色，并可跨多个项目参与。

### 4.5 内部经营财务台账

- 应收、收款、退款/冲销。
- 应付、付款、扣款。
- 销项发票和进项发票。
- 现金/银行账户和内部转账。
- 员工工资、报销、借支、分红/利息等内部往来。

本模块只做内账经营台账，不做完整会计核算。

### 4.6 员工与工资

- 正式员工和劳务员工。
- 员工部门、岗位、项目和班组归属历史。
- 工资批次和工资项目。
- 多方式、分批、多人收款的实际支付。
- 简单报销、借支和其他员工往来。

### 4.7 附件、导入导出、备份和提醒

- 统一附件上传、预览、下载和删除。
- 老系统数据迁移。
- 标准 Excel 导入导出。
- SQL Server 数据库和附件备份。
- 首页提醒中心和风险提示。

## 5. 组织与权限设计

### 5.1 组织范围

系统是单公司使用，但支持多个部门、分支机构和我方签约公司。不同我方签约公司不是独立租户，而是同一项目下可独立核算的法律主体。

### 5.2 管理员层级

#### 系统级管理员

- 拥有全部系统和业务权限。
- 管理用户、角色模板、数据范围、系统参数、备份恢复、附件策略和全局配置。
- 可以直接查看、修改和删除全部业务数据。
- 高风险操作必须记录完整审计。

#### 应用级管理员

- 管理普通用户、项目和业务基础资料。
- 管理状态、字典、导入导出模板和日常数据。
- 不能任命系统级管理员。
- 不能修改全局安全配置、备份恢复策略或系统级权限边界。

### 5.3 普通角色

- **财务人员**：管理内部经营台账、发票、工资和报销等财务模块。
- **项目负责人**：查看和维护自己负责或参与的项目、合同、阶段成果和项目附件。
- **现场人员**：只查看被分配的项目，提交阶段成果资料。
- **查询人员**：按授权范围只读查看和导出。
- **供应商/施工班组**：第一阶段不创建登录账号。

### 5.4 权限实现原则

权限由“角色模板 + 数据范围 + 用户覆盖”组成，不采用不可解释的单一数字等级。

系统管理员可以针对单个用户增加或撤销菜单、查看、新增、编辑、删除、导入、导出、备份和恢复权限。

## 6. 核心数据模型

以下是设计层面的核心实体，具体表名和字段在实施计划中再确定。

### 6.1 组织和主体

- `OrganizationUnit`：部门、分支机构。
- `LegalEntity`：我方签约公司。
- `BankAccount`：银行、现金和其他账户。
- `InvoiceProfile`：开票资料、税务信息和默认发票设置。

### 6.2 合作单位

- `BusinessPartner`：统一合作单位档案。
- `BusinessPartnerRole`：甲方、班组、材料供应商、零星供应商等角色。
- `PartnerContact`：联系人、电话、邮箱和地址。

### 6.3 项目合同

- `Project`：项目主档。
- `ProjectAssignment`：用户、员工或班组的项目分配。
- `Contract`：甲方合同、对外合同、补充协议和变更单。
- `ContractParty`：合同参与方。
- `ContractLegalEntityAllocation`：我方公司分摊。
- `ContractLineItem`：工程、材料或服务清单项。
- `ProjectMilestone`：简化项目节点计划。
- `StageResult`：阶段成果提交。
- `StageResultLine`：阶段成果关联清单项。

### 6.4 财务台账

- `ReceivableEntry`：应收明细。
- `CollectionEntry`：收款明细。
- `RefundOrReversalEntry`：退款、冲销和负数调整。
- `PayableEntry`：应付明细。
- `PaymentEntry`：付款明细。
- `DeductionEntry`：扣款。
- `InvoiceEntry`：销项或进项发票。
- `AccountTransaction`：账户流水。
- `AccountTransfer`：内部转账。

### 6.5 员工与工资

- `Employee`：正式或劳务员工档案。
- `EmployeeAffiliationHistory`：部门、项目、班组归属的时间历史。
- `PayrollBatch`：工资批次。
- `PayrollItem`：工资项目明细。
- `PayrollPayment`：实际支付记录。
- `ExpenseRecord`：简单报销记录。
- `EmployeeAdvance`：借支、备用金和归还。
- `EmployeeOtherPayment`：分红、利息或其他员工往来支付。

股权和利息分红第一阶段只登记每次应付/支付及附件，不保存持股比例、转让和股权变更历史。

### 6.6 通用实体

- `Attachment`：附件元数据。
- `AuditLog`：审计日志。
- `ImportBatch`：导入批次和错误记录。
- `BackupTask`：备份和恢复任务。
- `ReminderItem`：系统内提醒。

## 7. 项目、合同与工程量设计

### 7.1 项目主档

项目主档至少包括：

- 项目编号、名称、母案/总包信息。
- 甲方/总包单位和联系人。
- 项目负责人、部门、分支机构和关联我方公司。
- 项目阶段、财务状态和归档状态。
- 暂估工程金额、合同金额汇总、结算金额汇总。
- 项目节点计划。
- 项目附件和阶段成果入口。

### 7.2 多合同与共同签约

一个项目可以关联：

- 多份甲方合同。
- 多份补充协议和变更单。
- 多份施工班组、材料供应商和零星供应商合同。

通常一份合同只对应一家我方公司，但允许极少数合同由 2～3 家我方公司共同签署。

共同签署时支持：

- 按工程清单项分摊。
- 按固定金额分摊。
- 按比例分摊。

系统校验各公司分摊合计等于合同金额；每家公司仍分别保留应收、收款、发票和账户流水。

### 7.3 暂估与结算

每条清单项保存两组量价：

- 暂估工程量、暂估单价、暂估金额。
- 结算工程量、结算单价、结算金额。

公式：

```text
暂估金额 = 暂估工程量 × 暂估单价
结算金额 = 结算工程量 × 结算单价
```

项目汇总金额全部由清单项汇总产生，不允许手工修改总额。

清单项可以分别结算，因此项目状态可以是：

- `暂估`：没有清单项完成最终结算。
- `部分结算`：部分清单项使用结算量价，其他清单项仍使用暂估量价。
- `已结算`：项目清单项均完成最终结算。

部分结算时当前工程金额为：

```text
已结算清单项结算金额 + 未结算清单项暂估金额
```

### 7.4 项目状态维度

项目状态拆为三个维度：

1. **项目阶段**：前期、待签合同、待进场、施工中、暂停施工、完工待验收、结算中、质保期、已结束。
2. **财务状态**：由应收、收款、应付、付款和发票明细自动汇总。
3. **归档状态**：未归档、待归档、已归档。

不把施工阶段、收款状态和归档状态混在一个状态字段中。

### 7.5 简化节点计划

第一阶段不做完整甘特图，只记录：

- 计划开工、实际开工。
- 计划进场、实际进场。
- 关键阶段节点。
- 完工、验收、结算和归档日期。

## 8. 阶段成果与完工资料

施工日志不作为高频日记功能建设。系统提供阶段成果记录，用于阶段性成果、分部分项完成、验收和完工资料上传。

每次阶段成果包括：

- 阶段名称、日期和说明。
- 关联项目、合同和清单项。
- 本期完成量、累计完成量、剩余量。
- 完成比例和验收/质量情况。
- 阶段资料、照片和附件。
- 提交人、提交时间和修改记录。

手机端支持阶段成果草稿、工程量临时录入和照片保存；财务结算、应收确认、发票和付款必须在线操作。

## 9. 合作单位与对外业务

### 9.1 统一档案

一个合作单位允许拥有多个角色，例如同时是材料供应商和零星供应商。界面按业务角色提供独立入口，底层档案保持统一，避免重复录入。

### 9.2 跨项目视图

合作单位详情页必须支持：

- 参与项目列表。
- 每个项目的合同、清单量价、阶段完成量。
- 应付、已付、未付和扣款。
- 进项发票和附件。
- 单位总览汇总。

施工班组和供应商可以被复制为新档案，但不复制历史项目、流水、附件和唯一身份信息。

## 10. 内部经营财务台账

### 10.1 设计原则

财务模块是内部经营台账，目标是“账理清、来源清、汇总清、能追溯”。不建设完整会计报表系统。

### 10.2 应收与收款

应收款通过独立的应收记录产生，来源可以是：

- 合同付款节点。
- 阶段结算。
- 用户人工录入的经营应收。

合同金额不自动等于应收款。这样可以处理预付款、进度款、质保金、扣款和分公司分摊。

收款支持：

- 多次收款。
- 不同账户收款。
- 退款和负数冲销。
- 关联合同、应收记录、发票和项目。

### 10.3 应付与付款

班组、材料供应商和零星供应商的应付通过结算、合同或人工应付记录产生。

付款支持：

- 分批付款。
- 扣款。
- 不同我方公司和账户付款。
- 付款凭证附件。

### 10.4 账户与内部转账

每家我方签约公司可以拥有多个：

- 银行账户。
- 现金账户。
- 其他内部经营账户。

每笔收款、付款、发票和内部转账选择具体签约公司和账户。内部转账同时生成转出和转入记录，避免余额失真。

### 10.5 发票台账

发票使用独立明细模型，区分：

- 我方开出的销项发票。
- 我方收到的进项发票。

每张发票至少关联：签约公司、合同、清单项/应收记录、合作单位、发票号码、日期、类型、税率、不含税金额、税额、含税金额、状态和附件。

支持部分开票、一张发票关联多个清单项，以及按项目、合同、签约公司和单位汇总。

### 10.6 统一经营指标

```text
项目合同总额 = 项目合同清单项金额之和
项目暂估总额 = 暂估清单项金额之和
项目结算总额 = 结算清单项金额之和
已收款 = 收款合计 - 退款/冲销合计
未收款 = 应收合计 - 已收款
已开票 = 关联发票合计
未开票 = 可开票应收 - 已开票
已付款 = 付款合计 - 付款冲销合计
未付款 = 应付合计 - 已付款
```

超过应收/应付的收款或付款不直接阻止，但显示经营风险提示。

## 11. 员工、工资和内部往来

### 11.1 员工类型

- 正式员工。
- 劳务员工。

劳务员工可以由公司直接聘用，也可以归属施工班组。归属、结算对象和付款对象按时间段保存，允许变更并保留历史。

### 11.2 员工关联

员工可以跨多个项目工作，并关联：

- 部门和岗位。
- 项目。
- 施工班组。
- 我方签约公司。
- 工资成本分摊。

### 11.3 工资批次

支持：

- 自然月。
- 任意日期区间。
- 项目阶段。
- 工程节点。
- 临时结算批次。

### 11.4 工资项目

同一员工、同一批次可以组合：

- 固定月薪。
- 日薪或小时工资。
- 按工程量计件。
- 阶段包干金额。
- 加班、奖金和补贴。
- 借支、请假、罚款和其他扣款。
- 补发和冲减。

### 11.5 实际支付

一笔应付工资与实际支付分开保存。支持：

- 分多次支付。
- 现金、银行、微信、支付宝或其他方式。
- 员工本人、班组负责人或受托收款人。
- 每次支付的账户、日期、金额、经办人和凭证。

系统自动计算应发、已发、未发和借支抵扣。

### 11.6 报销与其他往来

报销采用简单单笔模型，至少保存：

- 员工、项目、部门和签约公司。
- 日期、费用类别、金额、支付方式和说明。
- 发票、收据和其他附件。
- 应付、已付、未付、退回和冲销状态。

分红、利息和其他员工往来只登记每次应付/支付，不建立完整股权或投资档案。

## 12. UI、统计和图形化展示

### 12.1 统一页面结构

主要页面采用：

```text
筛选区 → 指标卡 → 图形摘要 → 明细表格 → 分页/导出
```

详情页采用：

```text
核心信息 → 图表与进度 → 时间线 → 明细 → 附件与审计
```

### 12.2 项目图形化

- 项目数量和阶段分布。
- 合同、暂估、结算金额对比。
- 应收、已收、未收和收款比例。
- 应开票、已开票和未开票。
- 工程量完成进度和阶段成果时间线。
- 多签约公司金额对比。

### 12.3 台账图形化

- 按月收入、支出、收款、付款和工资趋势。
- 各账户余额和资金流向。
- 应收/应付对比。
- 单位、班组和员工成本分布。

### 12.4 手机端

- 先显示指标卡、风险提示和待同步草稿。
- 复杂表格采用横向滚动或分段显示。
- 阶段成果、照片和附件操作优先优化。
- 财务明细保留必要字段，避免手机端录入过度拥挤。

### 12.5 视觉和可访问性

- 统一简体中文文案。
- 颜色表达状态，进度条只表达完成度。
- 关键指标同时提供文字，不能只依赖颜色。
- 支持键盘操作、合理颜色对比和减少动效。
- 图形使用项目内置 SVG/HTML/CSS，不依赖外部 CDN。

## 13. 智能复制

项目、施工班组、供应商、员工和其他适合的主档支持智能复制。

复制流程：

1. 用户选择原记录并进入复制页面。
2. 系统预填可复用字段。
3. 用户选择复制范围并修改信息。
4. 系统自动清除唯一标识、隐私字段、银行账户、登录账号、历史流水、工资记录、附件和审计记录。
5. 保存为新记录。

可复制示例：

- 项目：清单结构、常用节点、默认配置。
- 班组/供应商：分类、工种、计价规则和结算设置。
- 员工：部门、岗位、员工类型、计薪方式和常用工资项目。

## 14. 导入、导出、备份与附件

### 14.1 老系统数据平移

针对用户提供的老系统 Excel、CSV 或其他导出文件编写一次性迁移工具：

- 平移项目、合同、合作单位、员工和财务流水。
- 迁移前检查必填字段、重复数据和关联关系。
- 生成预览和错误清单后正式导入。
- 保留原始文件和迁移批次。

不建设复杂通用字段映射平台。

### 14.2 日常 Excel

主要模块提供标准模板导入和导出：

- 项目、合同和工程清单。
- 合作单位和施工班组。
- 员工和工资。
- 收款、付款、发票和账户流水。
- 阶段成果和工程量。

### 14.3 自由导出

每个主要导出页面支持：

- 自由选择导出字段。
- 调整字段顺序。
- 设置筛选、排序和统计截止日。
- 自动保存当前用户上一次设置。
- 保存多个个人导出模板。
- 系统管理员发布共享模板。

项目经营总览默认导出：

1. `总览汇总`：项目数量、合同总额、应收、已收、未收、应开票、已开票、未开票、比例等。
2. `项目明细`：每个项目一行，包含基础信息和经营指标。
3. 可选明细页：合同、收款、发票、工程量和阶段成果。

导出设置按用户和页面保存，领导常用模板由管理员发布。

### 14.4 简单完整备份

- 定时或手动备份 SQL Server 数据库。
- 同步备份附件目录。
- 支持数据库和附件整体迁移到新服务器。
- 提供清晰的备份、下载和恢复操作。
- 不做复杂的分模块冲突合并。

### 14.5 附件

附件支持：

- 上传、预览、下载和删除。
- 分类和说明。
- 关联项目、合同、阶段成果、财务、员工等记录。
- 上传人、上传时间和文件基本信息。
- 按业务对象继承访问权限。

第一阶段不做复杂的版本分支和在线协同编辑。

## 15. PWA 与有限离线

### 15.1 离线范围

仅支持：

- 阶段成果草稿。
- 工程量临时录入。
- 备注。
- 照片和阶段资料。

不支持离线提交：

- 收款、付款、发票。
- 工资结算。
- 账户流水。
- 合同金额和最终结算。

### 15.2 同步规则

- 本地保存草稿和同步队列。
- 每条草稿具有客户端标识和版本号。
- 同步成功后标记为已同步。
- 图片可以单独重试。
- 服务器记录已被修改时不自动覆盖，提示用户比较和确认。
- 同步失败在首页提醒中心显示。

## 16. 提醒中心

第一阶段只做系统内提醒和首页风险提示，包含：

- 项目节点临近或逾期。
- 应收未收。
- 应付未付。
- 应开票未开票。
- 工资未发。
- 离线草稿待同步或同步失败。
- 导入、导出、备份失败。

不在第一阶段接入邮件、短信、企业微信或其他外部通知。

## 17. 审计、安全与删除

### 17.1 审计内容

所有新增、修改、删除、恢复、导入、导出、备份、恢复和权限调整记录：

- 操作人。
- 时间。
- 设备/IP和请求标识。
- 业务对象及关联项目。
- 修改原因。
- 修改前完整值。
- 修改后完整值。
- 附件变化。

### 17.2 删除规则

拥有权限的用户可以直接修改或删除财务和业务记录，但系统使用逻辑删除保留历史，敏感操作必须填写原因。

删除前检查关键关联；存在关联时优先提示改为停用或冲销，避免破坏汇总和历史。

### 17.3 数据安全

- 密码使用 ASP.NET Core Identity 安全存储。
- 附件不通过公开目录直接暴露，下载时再次检查业务权限。
- 导出文件按用户权限和筛选条件生成。
- 备份文件和附件目录应限制服务器账户访问。
- 生产环境使用 HTTPS、最小权限账户和定期备份。

## 18. 校验、异常和并发

### 18.1 校验

- 合同公司分摊必须等于合同金额。
- 暂估和结算总额只能由清单项计算。
- 金额、数量、税率和日期格式正确。
- 收款/付款超过应收/应付时提示风险而非强制阻止。
- 关联记录不存在时不得保存。
- 导入错误显示文件、工作表、行号、字段和原因。

### 18.2 并发

项目、合同、清单、财务记录和工资批次使用行版本或等价机制防止覆盖。

### 18.3 异常

- 页面使用简体中文错误信息和处理建议。
- 未预期异常记录技术日志和请求标识。
- 关键写入使用事务。
- 后台任务失败进入提醒中心。

## 24. 分阶段开发策略

由于系统功能较多，采用分阶段交付，不要求一次性完成全部模块。阶段边界按业务依赖和可验证成果划分，避免同一数据模型被多个阶段反复重写。

### 24.1 阶段划分

#### 阶段 0：项目基线

- 初始化新项目目录和 Git 仓库（需用户确认新仓库/远程信息后执行）。
- 建立 ASP.NET Core、SQL Server、IIS 本地运行基线。
- 建立基础分层、迁移、测试、日志、备份目录和进度文档。
- 搭建简体中文 UI 外壳、响应式布局、基础组件和 PWA 外壳。

#### 阶段 1：身份、组织与权限

- 登录、用户、部门、分支机构和我方签约公司。
- 系统级管理员、应用级管理员和基础角色。
- 数据范围、用户个别权限和审计基础。

#### 阶段 2：项目、合同与工程清单

- 项目主档、合作单位基础关联。
- 多合同、补充协议、变更单和工程清单。
- 我方公司共同签署与金额/比例/清单项分摊。
- 暂估量价、结算量价和项目金额计算。
- 项目阶段、归档状态和简化节点计划。

#### 阶段 3：合作单位与阶段成果

- 统一合作单位档案和角色视图。
- 施工班组、材料供应商、零星供应商跨项目视图。
- 阶段成果、工程量累计、完工资料和附件基础能力。

#### 阶段 4：内部经营财务台账

- 应收、收款、退款/冲销。
- 应付、付款、扣款。
- 销项/进项发票。
- 现金/银行账户和内部转账。
- 项目、签约公司、合作单位和账户汇总图表。

#### 阶段 5：员工、工资与员工往来

- 正式员工、劳务员工和归属历史。
- 工资批次、组合工资项目、项目成本分摊。
- 多次、多渠道、多收款人的实际支付。
- 简单报销、借支、分红/利息和其他往来。

#### 阶段 6：导入、导出、备份与提醒

- 老系统样例数据迁移工具。
- 主要模块 Excel 模板导入导出。
- 筛选字段、导出汇总页、项目明细页和个人/共享模板。
- SQL Server + 附件整体备份。
- 首页提醒中心和风险提示。

#### 阶段 7：PWA 有限离线与整体图形化

- 阶段成果草稿离线保存。
- 工程量、备注和照片同步。
- 版本冲突提示、失败重试和待同步提醒。
- 电脑端和手机端图表、卡片和响应式优化。

#### 阶段 8：集成验收与部署

- 全模块联动验证。
- 关键经营台账与领导导出样例验收。
- 权限、安全、备份恢复、性能和浏览器测试。
- Windows Server + IIS 部署和上线检查。

### 24.2 阶段执行原则

- 每个阶段先读取本设计文档和唯一进度文档，不重复进行完整需求梳理。
- 每阶段开始前只补充本阶段必要的局部确认。
- 每阶段完成后必须运行与本阶段相关的测试和验证。
- 发现设计偏差时，先更新设计文档的决策记录，再修改实现，避免代码和文档分叉。
- 阶段之间优先复用已经稳定的实体、组件和测试，不重复搭建基础设施。
- 跨阶段公共能力（审计、附件、权限、导出、金额类型、分页、提醒）在首次需要时建立可复用基础，后续只扩展。

## 25. 唯一开发进度文档

开发过程中只维护一个进度文件：

`docs/开发进度.md`

每次阶段开发完成后，在同一个文件中更新“当前状态”，并在“阶段历史”追加一条记录，不为每个阶段新建独立进度文件。

进度文档必须持续包含：

- 当前阶段、总体完成比例和更新时间。
- 已完成模块和可验证功能。
- 本阶段修改/新增的文件和数据库迁移。
- 已执行的构建、测试、浏览器和备份恢复验证。
- 当前已知问题、暂未实现项和风险。
- 与设计文档相比发生的偏差或新增决策。
- 下一阶段目标、前置条件和建议执行顺序。
- 全局阶段路线图和已完成阶段标记。

每次新的开发任务开始时，先读取：

1. 本详细设计文档。
2. `docs/开发进度.md` 的当前状态和下一步计划。
3. 与当前阶段直接相关的代码、测试和项目文档。

只有当用户明确改变需求时，才重新进入完整设计讨论；普通阶段推进不重复执行已经确认的需求流程。

## 26. 设计文档确认后的工作流

1. 用户确认本详细设计文档。
2. 编写对应阶段的实施计划。
3. 用户确认实施计划后，初始化新项目和 Git 仓库。
4. 按阶段实施，每阶段完成后更新 `docs/开发进度.md`。
5. 阶段验证通过后，再进入下一阶段。
6. 所有阶段完成后，进行整体验收和部署，不将未完成阶段伪装为已完成。
