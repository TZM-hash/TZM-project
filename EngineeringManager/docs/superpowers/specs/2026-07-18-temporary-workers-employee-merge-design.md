# 临时人员并入员工管理设计

**日期：** 2026-07-18
**状态：** 已确认并完成实现、测试库迁移验证和质量门禁

## 1. 目标

取消独立的临时人员主档和页面，将公司直接聘用的临时人员作为员工管理中的特殊人员统一管理，并完整保留历史工资付款、项目归属、敏感资料和审计关系。

## 2. 核心模型

扩展员工类型：

```text
EmployeeType
- Formal      正式员工
- Labor       劳务员工
- Temporary   特殊临时人员
```

特殊临时人员与其他员工共用：

- 员工主档。
- 项目、班组、部门和自有公司归属历史。
- 工资批次和工资付款。
- 员工年度总账、领款和调整。
- 敏感资料权限和修改审计。
- 可选证书和其他员工扩展能力。

人员类型可以在有权限的编辑流程中从特殊临时人员变更为劳务员工或正式员工，不复制人员主档。

## 3. 数据迁移

### 3.1 已关联员工

`TemporaryWorker.ConvertedEmployeeId` 已存在时，临时人员视为该员工的历史临时身份：

- 不新建员工。
- 将临时人员历史工资付款迁移到该员工。
- 保留原员工当前类型，不强制改为 `Temporary`。
- 临时人员资料只补充原员工为空的字段；发生冲突时进入迁移报告，不覆盖已有员工资料。

### 3.2 未关联员工

未关联员工的临时人员创建新的 `Temporary` 类型员工：

- 生成稳定且唯一的员工编号。
- 姓名、身份证、电话、银行卡、开户行、工种、备注和启用状态原样迁移。
- 工种映射到员工岗位名称。
- 默认项目转换为一条主要项目归属记录，起始日期使用原临时人员创建日期。
- 无法确定的公司、部门或班组归属保持为空，不擅自推断。

### 3.3 重复与冲突

迁移前生成待处理报告，至少检测：

- 身份证与已有员工冲突。
- 一个员工被多个临时人员档案引用。
- 同名、同电话或同银行卡的疑似重复人员。
- 临时人员历史付款缺少有效人员引用。
- 员工编号生成冲突。

明确的 `ConvertedEmployeeId` 优先级最高。没有明确关联但身份证完全相同的记录不得静默合并，必须在迁移前确认或通过受控规则处理。

### 3.4 历史工资付款

现有 `PayrollPayment` 中临时人员付款迁移为员工付款：

- `EmployeeId` 指向合并后的员工。
- 收款对象类型改为员工。
- 原姓名、身份证、电话、银行卡、工种和收款人快照继续保留。
- 工资批次、项目、账户、日期、金额和账户流水引用保持不变。
- 迁移后员工年度总账能够读取这些历史付款。

迁移完成后，临时人员专用外键和工资收款对象分支不再用于新数据。确认所有引用已迁移后，删除独立临时人员业务表和工资专用字段，同时保留只含旧临时人员 ID、新员工 ID 和迁移时间的 `PersonnelMigrationMap`，用于旧链接跳转和审计倒查。该映射不是可编辑人员主档，也不参与业务统计。

## 4. 应用行为

### 4.1 员工管理

- 员工列表新增特殊临时人员类型、筛选和计数。
- 员工总数分别显示正式员工、劳务员工、特殊临时人员和在用员工。
- 新增或编辑员工时可以选择特殊临时人员。
- 特殊临时人员使用统一员工详情、年度总账、工资、领款、归属和修改历史。
- 类型变更只更新员工类型，不复制历史数据。

### 4.2 工资管理

- 工资批次不再从独立临时人员表读取。
- 特殊临时人员与其他员工统一选择，并支持按员工类型筛选。
- 原临时工资批次类型可以继续作为批次业务分类，但收款对象统一为员工。
- 工资统计按员工类型分组时，特殊临时人员单独列示。

### 4.3 年度总账

迁移后的特殊临时人员自动使用现有员工年度总账：

- 历史工资付款进入已领金额。
- 后续工资、领款和调整进入对应业务年度。
- 跨年度余额按现有员工年度规则结转。
- 员工详情可以查看迁移前后的完整连续历史。

## 5. 页面与导航

- 删除独立“临时人员”主导航。
- 删除临时人员新增、列表和详情作为独立业务入口。
- 旧 `/TemporaryWorkers/...` 链接根据迁移映射跳转到对应员工详情；无法映射时返回明确的历史迁移提示。
- 工资编辑页移除独立临时人员区，使用统一员工表和类型筛选。
- 员工列表和详情显示“特殊临时人员”标签，但不降低其可用能力。

## 6. 权限

- 临时人员读取权限并入员工读取权限。
- 临时人员管理权限并入员工管理权限。
- 敏感字段继续使用现有敏感人员资料权限。
- 迁移期间保留旧权限到新权限的兼容映射，完成角色核对后删除旧权限键。

## 7. 审计与兼容

- 保存每个旧临时人员 ID 到新员工 ID 的迁移映射。
- 迁移操作记录来源、执行时间和合并规则。
- 已关联员工的字段冲突记录在迁移报告中。
- 历史 `AuditLog` 不改写；查看时通过迁移映射定位新员工。
- 导入、导出、样例数据、服务注册、PWA 敏感路径和导航均移除独立临时人员概念。

## 8. 实施顺序

1. 增加 `EmployeeType.Temporary` 及员工页面、导入导出和显示支持。
2. 增加迁移映射和迁移前冲突检查。
3. 创建或匹配员工，并迁移人员资料和项目归属。
4. 迁移工资付款外键和收款对象类型。
5. 更新工资服务、年度总账和页面读取逻辑。
6. 增加旧链接跳转并移除独立导航。
7. 验证无残留引用后移除临时人员服务、DTO、页面和专用权限。
8. 最后移除临时人员表、工资专用外键及不再使用的收款对象枚举分支。

数据库迁移必须在测试库先执行，并在删除旧结构前验证人员数、付款笔数、付款金额、账户流水和年度余额一致。

## 9. 测试与验收

至少覆盖：

- 未关联临时人员创建特殊临时员工。
- 已关联临时人员合并到原员工且不重复建档。
- 字段冲突不覆盖已有员工资料。
- 身份证和员工编号冲突进入待处理报告。
- 默认项目转换为员工主要归属。
- 历史工资付款全部改挂员工且金额不变。
- 付款快照、批次、项目、账户和流水关系不变。
- 特殊临时人员出现在员工列表、筛选、工资和年度总账。
- 特殊临时人员变更为正式或劳务员工后历史连续。
- 旧临时人员链接正确跳转。
- 独立导航、服务和权限移除后无残留入口。
- 迁移前后人员、付款和年度余额汇总一致。

验收时必须输出迁移对账结果：临时人员总数、合并到已有员工数、新建特殊临时员工数、冲突数、迁移付款笔数和迁移付款总额。
