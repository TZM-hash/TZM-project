# 项目业务明细快捷编辑与财务入口精简设计

## 1. 背景

项目详情页当前同时提供快捷编辑和独立的“详细编辑”页面。实际使用中，用户希望所有日常修改都在项目详情页的快捷编辑完成，并删除独立详细编辑页及其入口。与此同时，收款、开票、付款和施工详情需要统一新增附件操作，收款方式需要支持自定义文字，项目应收金额必须以工程量明细汇总为准。

## 2. 已确认目标

1. 删除所有标签页的“详细编辑”入口及 `Projects/Records/Edit` 页面，不再提供项目详情到独立编辑页的跳转。
2. 快捷编辑覆盖日常业务记录需要修改的字段，并继续使用现有项目权限、并发校验、审计和中央账本事务。
3. 收款记录删除“关联应收”选择功能。后台按项目工程量自动生成的应收记录进行收款分摊，前台不展示应收选择器。
4. 收款方式改为自定义文字，原样保存、显示和编辑；收款账户、收款金额、收款方式使用明确中文标签。
5. 发票明细固定为销项发票，删除“方向”列，保留现有日期、号码、合同、开票公司、往来单位、税率、发票类型和金额字段。
6. 应付记录和实际付款记录保持现有明细字段，不增加与业务无关的展示字段；两类记录均支持附件。
7. 收款、发票、应付、付款和施工记录每条只保留一个有效附件，再次上传替换原附件。
8. 施工记录将跨项目流转、关联后续项目和主体身份变更从普通快捷编辑中分离为受控操作。
9. 中央账本精简项目相关人工录入入口，但保留历史数据、查询、审计、对账和管理员修正能力。

## 3. 非目标

- 不删除历史财务记录、附件、审计日志或中央账本表结构。
- 不把工程量应收改回人工维护；工程量明细仍是应收金额唯一业务来源。
- 不改变其他模块（工资、合作商、施工班组等）已有财务规则。
- 不引入异步单页应用或新的前端框架，继续使用 Razor Pages 和现有快捷编辑交互。

## 4. 页面与交互

### 4.1 项目详情页

- 移除收款、发票、付款、施工详情标签页顶部的“详细编辑”按钮。
- 删除所有指向 `Projects/Records/Edit` 的链接、文案和测试断言。
- 删除 `Projects/Records/Edit.cshtml`、对应页面模型及仅供该页使用的编辑 partial；相关日常编辑能力迁移到项目详情页现有快捷编辑表格。
- 保留“快捷编辑”按钮、逐行保存、取消编辑、修改原因和当前标签页重定向行为。

### 4.2 工程量明细

- 延续现有新增工程量折叠区域和行级快捷编辑。
- 继续支持每条工程量一个附件，上传新文件替换旧文件。
- 工程量金额为 `工程量 × 单价`，工程量明细汇总作为项目应收金额来源。

### 4.3 收款明细

页面只保留“应收汇总”和“收款记录”两部分，不再显示应收记录表格，也不提供新增或编辑应收的入口。

收款新增和快捷编辑字段：

- 收款日期
- 合同
- 签约公司
- 付款方
- 收款账户
- 收款金额
- 收款方式（自由文本）
- 备注
- 附件

删除字段：

- 业务类型中的“新增应收”选项
- 关联应收
- 收款记录中的手工应收编辑入口

收款方式不能再绑定 `PaymentMethod` 枚举。收款相关应用层 DTO、项目工作区 DTO、导入导出和显示逻辑改为可空字符串，并保留旧枚举值的历史文本兼容；付款记录仍沿用现有付款方式规则，不随收款方式一起改动。

### 4.4 收款自动分摊

前台不再让用户选择应收记录。后台创建或修改收款时，按当前项目工程量自动生成的有效应收记录执行自动分摊：

1. 先按最早到期日排序。
2. 到期日相同按最早业务日期排序。
3. 再按稳定主键排序，确保重复请求得到一致结果。
4. 收款可以分摊到多条工程量应收。
5. 收款超过当前有效应收时，已分摊部分计入项目已收汇总，剩余金额保留为中央账本未分摊余额。收款记录仍必须归属于发起收款的项目，未分摊余额在中央账本查询中可见。

为支持“没有可分摊余额但仍属于当前项目”的场景，在 `LedgerSourceType` 中增加项目收款来源值，并将项目 ID 保存到收款记录的 `SourceId`。项目收款查询先读取该来源，再叠加收款分摊金额；跨项目中央账本记录仍保持原有分摊模型。

该规则保持现有 `FinanceCashAllocation`、项目收款查询和财务汇总的一致口径，同时去掉用户手工选择关联应收的负担。若项目没有任何工程量生成的有效应收，拒绝创建收款并提示先完善工程量；若存在工程量应收但剩余可收金额为零，允许创建项目归属的全额未分摊收款。

### 4.5 开票明细

- 固定为销项发票，不再显示方向字段或方向编辑控件。
- 保留日期、发票号码、合同、开票公司、往来单位、税率与发票类型、不含税金额、税额、含税金额和状态。
- 往来单位只显示一次，不新增重复字段。
- 快捷编辑需要覆盖当前表格展示的可编辑字段；合同、开票公司、往来单位和税率不能因为隐藏字段而丢失编辑能力。
- 新增和已有发票记录均提供一个附件入口。

### 4.6 付款明细

应付记录和实际付款记录保留当前明细页已有字段，不为精简而改变业务展示口径：

- 应付记录继续显示日期、到期日、合同、签约公司、收款单位、金额、说明、状态。
- 实际付款继续显示付款日期、合同、签约公司、收款单位、付款账户、付款金额、付款方式、备注。
- 两类记录各自支持一个附件；附件列采用紧凑的预览、替换和删除操作，不扩展出中央账本分摊字段。
- 工资代发形成的付款记录继续显示来源批次，只允许查看来源和管理附件，不允许从项目快捷编辑直接改写工资批次派生字段。

### 4.7 施工详情

普通快捷编辑保留日期、停工天数、备注、项目总览显示等记录字段。设备/施工班组类型和主体在正式记录创建后锁定，避免历史记录身份被改写。

跨项目操作改为三个受控动作：

- `关联后续项目`：选择目标项目和进场日期，服务端在一个事务内创建或复用后续草稿，并同时设置当前记录的 `NextRecordId`、后续记录的 `PreviousRecordId` 及双方项目字段。
- `解除流转`：在一个事务内解除当前记录与后续记录的双向关系，并清空双方对应项目字段。
- `连接上一条记录`：按同一设备或班组、时间顺序查找可连接记录，用户确认后完成双向连接。

每个动作都检查当前记录和对端记录的并发版本、主体一致性和循环关系。连接失败时整个事务回滚。已存在正式记录时不允许直接切换设备/班组；需要纠正时，新建正确记录并将原记录保留为历史修正对象。

## 5. 中央账本精简

- 移除项目详情页跳转到中央账本的“详细录入”按钮。
- 移除项目场景中的人工新增应收入口；工程量同步服务是项目应收唯一创建来源。
- 移除项目收款表单的关联应收选择和手工分摊入口。
- 保留中央账本全局查询、历史记录、审计、对账、未分摊余额查看和管理员级修正能力。
- 保留发票、应付、付款的账本记录及其必要的全局分摊能力；项目页只暴露当前项目日常编辑需要的字段。
- 不删除 `FinanceSettlement`、`FinanceCashEntry`、allocation 或审计表，避免破坏既有数据、导入导出和对账结果。

## 6. 附件设计

附件类型沿用现有项目业务附件模型：

- 收款：`Cash`
- 发票：`Invoice`
- 应付：`Settlement`
- 付款：`Cash`
- 施工：`Construction`

附件服务增加通用的单附件替换能力，替换顺序为先保存新文件，再在同一数据库事务中软删除旧附件并写入新附件。新文件校验或存储失败时，原附件保持有效。继续使用现有 20MB 限制、文件名校验、项目归属校验和管理权限。

## 7. 错误处理与权限

- 所有快捷编辑继续要求项目管理或财务管理权限，并保留防伪令牌。
- 所有财务更新继续通过现有服务完成，账户流水、分摊、汇总和审计在同一事务中更新。
- 并发冲突返回当前标签页并保留中文错误提示，要求用户刷新后重试。
- 收款自动分摊找不到任何工程量生成的有效应收时，拒绝写入并提示先完善工程量或签约公司；若有效应收存在但已全部分摊，允许创建带项目来源的全额未分摊收款，不生成无法归属项目的孤立收款。
- 施工双向连接的任一端校验失败时不得留下半连接或单向关系。

## 8. 验证策略

### Web 页面

- 所有标签页不再包含“详细编辑”按钮和 `Projects/Records/Edit` 链接。
- 旧详细编辑路由返回 404。
- 收款页不包含应收记录表格、人工新增应收入口或关联应收控件。
- 发票表格不包含方向列，往来单位只出现一次。
- 付款页字段保持现有展示，收款、发票、应付、付款、施工均包含附件控件。
- 快捷编辑可覆盖页面展示字段，隐藏字段不会导致数据无法修改。

### 应用服务

- 工程量新增或修改会同步更新自动应收金额。
- 收款自动按确定顺序分摊，多条应收和超额未分摊场景均有测试。
- 收款方式任意文字可创建、修改、显示并兼容旧值。
- 各业务记录单附件首次上传、替换、删除和失败回滚均有测试。
- 施工连接、解除连接、重复连接、并发冲突和循环保护均有测试。
- 工资代发付款不能通过项目快捷编辑改写派生记录。

### 构建与浏览器

- 运行目标 Web 测试、完整 `dotnet test` 和 `dotnet build`。
- 启动本地 Web 应用，检查桌面和窄屏下各标签页的快捷编辑、折叠新增、附件预览/替换、表格横向滚动和施工流转操作，不出现控件重叠或字段截断。

## 9. 验收标准

- 项目详情页是所有日常编辑的唯一入口，详细编辑按钮、页面和路由均不存在。
- 应收金额始终由工程量明细汇总产生，用户不能手工新增或编辑项目应收。
- 收款不再选择关联应收，但项目收款仍可按自动分摊规则进入项目汇总，超额部分可追踪。
- 收款方式支持任意文字。
- 发票只显示销项信息，方向列删除且往来单位不重复。
- 应付和付款保持现有明细字段，并且两类记录都支持单附件替换。
- 施工流转使用事务化专用动作，不允许普通编辑破坏双向关系、主体身份或并发一致性。
- 自动化测试、构建和浏览器验证通过。
