# 项目部署手册

本文说明 `EngineeringManager` 的三阶段部署路线：

1. 当前在开发电脑上进行本机测试。
2. 正式版仍部署在本机，由办公室其他电脑通过局域网访问。
3. 最终迁移到云服务器，由用户通过公网域名访问。

当前技术基线为 Windows、.NET 10、ASP.NET Core Razor Pages、IIS 和 SQL Server。本文中的 PowerShell 命令均使用 PowerShell 7，执行目录默认为项目根目录 `EngineeringManager`。

## 一、部署路线

| 阶段 | 应用运行方式 | 数据库 | 访问范围 | 当前建议 |
| --- | --- | --- | --- | --- |
| 本机测试 | 仓库内 `dotnet run` | `localhost\SQLEXPRESS / EngineeringManager_Test` | 仅本机 | 当前使用 |
| 局域网临时联机测试 | Kestrel 监听本机局域网网卡 | `EngineeringManager_Test` | 同一局域网 | 仅短期验收 |
| 局域网正式运行 | 本机 IIS + HTTPS | 正式库 `EngineeringManager` | 办公室局域网 | 正式版第一部署目标 |
| 云端正式运行 | Windows Server + IIS + HTTPS | 云端正式 SQL Server | 公网用户 | 最终部署目标 |

无论在哪个阶段，浏览器客户端都不直接连接 SQL Server，也不共享程序目录。客户端只访问 Web 地址，SQL Server 端口不向普通客户端或公网开放。

## 二、必须遵守的安全边界

- 测试数据库固定使用 `EngineeringManager_Test`，正式数据库不得带 `_Test` 后缀。
- 正式环境必须设置 `ASPNETCORE_ENVIRONMENT=Production`。
- 正式环境必须关闭 `DevelopmentSampleData__Enabled`，不得复制测试账号文件 `local-test-credentials.txt`。
- 程序启动不会自动执行 EF Migration。数据库升级必须由维护人员在备份后人工执行。
- 数据库、`App_Data/attachments` 附件目录和当前发布包必须一起备份。
- 生产数据库迁移必须再次明确确认；不得把测试数据库直接覆盖到正式数据库。
- 当前特殊临时人员合并迁移为不可逆迁移。正式库执行全部迁移前，必须保留可恢复的数据库完整备份。
- 工资、身份证、银行卡、合同和财务数据属于敏感信息，正式访问必须使用 HTTPS。

## 三、阶段一：当前本机测试

### 3.1 环境要求

- 项目路径：`D:\AI\TZM-project\EngineeringManager`。
- SQL Server Express 实例：`localhost\SQLEXPRESS`。
- 测试数据库：`EngineeringManager_Test`。
- 项目自带 PowerShell、.NET SDK 和 EF 工具，不依赖系统全局 SDK。

检查 SQL Server 服务：

```powershell
$ErrorActionPreference = 'Stop'
Get-Service -Name 'MSSQL$SQLEXPRESS' | Select-Object Status, DisplayName
```

检查项目 SDK：

```powershell
$ErrorActionPreference = 'Stop'
& .\.tools\pwsh\pwsh.exe -NoLogo -NoProfile -File .\scripts\dotnet.ps1 --version
```

### 3.2 重建测试数据库

首次运行、迁移变更后或需要恢复完整演示数据时执行：

```powershell
$ErrorActionPreference = 'Stop'
& .\.tools\pwsh\pwsh.exe -NoLogo -NoProfile -File .\scripts\reset-test-database.ps1 -DatabaseName 'EngineeringManager_Test'
```

该脚本只接受 Development 配置中的 `EngineeringManager_Test`，会删除并重建测试库、应用全部迁移并生成演示数据。不要把它改成正式数据库名运行。

### 3.3 启动程序

```powershell
$ErrorActionPreference = 'Stop'
& .\.tools\pwsh\pwsh.exe -NoLogo -NoProfile -File .\scripts\dotnet.ps1 run --project .\src\EngineeringManager.Web\EngineeringManager.Web.csproj --urls 'http://127.0.0.1:5075'
```

浏览器访问：

```text
http://127.0.0.1:5075
```

演示账号保存在 Git 忽略的文件中：

```text
src/EngineeringManager.Web/App_Data/local-test-credentials.txt
```

该文件只能用于本机测试，不得发送给局域网正式用户，也不得复制到发布目录。

### 3.4 健康检查

另开一个 PowerShell 窗口执行：

```powershell
$ErrorActionPreference = 'Stop'
Invoke-WebRequest -Uri 'http://127.0.0.1:5075/health/live' -UseBasicParsing | Select-Object StatusCode
Invoke-WebRequest -Uri 'http://127.0.0.1:5075/health/ready' -UseBasicParsing | Select-Object StatusCode
```

- `/health/live` 返回 `200`：应用进程正常。
- `/health/ready` 返回 `200`：应用和数据库连接均正常。

停止程序时，在运行窗口按 `Ctrl+C`。

## 四、阶段一补充：局域网临时联机测试

该方式适合正式部署前，让同一办公室内的少量人员临时体验测试数据。它不是长期正式部署方式。

### 4.1 网络准备

将服务器电脑的当前网络设置为“专用网络”，并查看局域网 IPv4 地址：

```powershell
$ErrorActionPreference = 'Stop'
Get-NetIPAddress -AddressFamily IPv4 |
    Where-Object { $_.IPAddress -notlike '127.*' -and $_.PrefixOrigin -ne 'WellKnown' } |
    Select-Object InterfaceAlias, IPAddress
```

假设本机地址为 `192.168.1.20`。

### 4.2 开放临时测试端口

使用管理员身份启动 PowerShell 7：

```powershell
$ErrorActionPreference = 'Stop'
New-NetFirewallRule `
    -DisplayName 'EngineeringManager 临时局域网测试' `
    -Direction Inbound `
    -Action Allow `
    -Protocol TCP `
    -LocalPort 5075 `
    -Profile Private
```

### 4.3 监听局域网地址

```powershell
$ErrorActionPreference = 'Stop'
& .\.tools\pwsh\pwsh.exe -NoLogo -NoProfile -File .\scripts\dotnet.ps1 run --project .\src\EngineeringManager.Web\EngineeringManager.Web.csproj --urls 'http://0.0.0.0:5075'
```

其他电脑使用浏览器访问：

```text
http://192.168.1.20:5075
```

### 4.4 临时方式的限制

- 只允许使用测试数据，不录入真实工资、身份证、银行卡、合同或财务数据。
- PowerShell 窗口关闭后服务立即停止，没有自动重启。
- 使用 HTTP 时，非 `localhost` 地址可能无法完整使用需要安全上下文的 PWA 能力。
- 不要在路由器中做端口映射，不允许从互联网访问 `5075`。
- 测试结束后关闭程序并删除防火墙规则：

```powershell
$ErrorActionPreference = 'Stop'
Remove-NetFirewallRule -DisplayName 'EngineeringManager 临时局域网测试'
```

需要长期多人使用时，必须切换到下一节的 IIS + HTTPS 正式部署。

## 五、阶段二：本机作为局域网正式服务器

### 5.1 推荐拓扑

```text
办公室电脑/手机
        |
   局域网 HTTPS 443
        |
固定 IP 的 Windows 服务器电脑
   |-- IIS / EngineeringManager
   |-- SQL Server
   |-- App_Data/attachments
   |-- App_Data/backups
```

初期可以让 IIS 和 SQL Server 运行在同一台电脑。客户端只访问 IIS，不开放 SQL Server 给客户端。

### 5.2 服务器电脑要求

- 建议使用 Windows 11 Pro 或 Windows Server，不使用家庭版作为长期服务器。
- 使用有线网络和固定内网 IP，或在路由器 DHCP 中绑定固定地址。
- 关闭自动休眠和自动断网，配置停电恢复；建议配备 UPS。
- 安装 SQL Server Express 或 Standard、IIS 和 .NET 10 ASP.NET Core Hosting Bundle。
- SQL Server Express 数据库上限为 10 GB，且没有 SQL Server Agent。数据规模接近限制或需要自动维护作业时，应升级为 SQL Server Standard。
- 系统盘、数据库盘和备份盘尽量分离，备份不能只保存在同一块硬盘。

### 5.3 安装 IIS

使用管理员 PowerShell 7 启用基础 IIS 功能：

```powershell
$ErrorActionPreference = 'Stop'
Enable-WindowsOptionalFeature `
    -Online `
    -FeatureName IIS-WebServerRole, IIS-WebServer, IIS-ManagementConsole, IIS-StaticContent, IIS-DefaultDocument, IIS-HttpErrors, IIS-RequestFiltering `
    -All
```

随后安装与 .NET 10 匹配的 ASP.NET Core Hosting Bundle，并重启电脑或执行：

```powershell
$ErrorActionPreference = 'Stop'
iisreset
```

在服务器上执行 `dotnet --list-runtimes`，确认存在 `Microsoft.AspNetCore.App 10.x`。

### 5.4 生成正式发布包

在开发目录执行：

```powershell
$ErrorActionPreference = 'Stop'
& .\.tools\pwsh\pwsh.exe -NoLogo -NoProfile -File .\scripts\quality-gate.ps1
```

必须看到：

```text
PUBLISH_RELEASE=PASS
QUALITY_GATE=PASS
```

发布包位于：

```text
artifacts/publish
```

### 5.5 正式目录规划

推荐目录：

```text
D:\EngineeringManager\
  app\                 IIS 当前运行目录
  releases\            每次发布包的只读副本
  backup\              升级前的程序和配置快照
```

首次部署：

```powershell
$ErrorActionPreference = 'Stop'
$root = 'D:\EngineeringManager'
New-Item -ItemType Directory -Path "$root\app", "$root\releases", "$root\backup" -Force | Out-Null
Copy-Item -Path '.\artifacts\publish\*' -Destination "$root\app" -Recurse -Force
New-Item -ItemType Directory -Path "$root\app\App_Data\attachments", "$root\app\App_Data\backups", "$root\app\App_Data\logs" -Force | Out-Null
```

升级时必须保留 `App_Data`，不能用空发布包覆盖附件、备份和日志目录。

### 5.6 正式数据库

正式数据库建议命名：

```text
EngineeringManager
```

连接字符串示例：

```text
Server=localhost\SQLEXPRESS;Database=EngineeringManager;Integrated Security=True;MultipleActiveResultSets=true;TrustServerCertificate=True
```

正式连接字符串不得写入 Git。建议通过 IIS 环境变量设置：

```text
ConnectionStrings__DefaultConnection
```

### 5.7 数据库迁移

迁移前先完成数据库和附件备份，然后在项目源码目录执行：

```powershell
$ErrorActionPreference = 'Stop'
$env:ASPNETCORE_ENVIRONMENT = 'Production'
$env:DOTNET_ROOT = (Join-Path (Get-Location) '.dotnet')
$env:PATH = "$env:DOTNET_ROOT$([IO.Path]::PathSeparator)$(Join-Path (Get-Location) '.tools\dotnet-tools')$([IO.Path]::PathSeparator)$env:PATH"
$env:ConnectionStrings__DefaultConnection = 'Server=localhost\SQLEXPRESS;Database=EngineeringManager;Integrated Security=True;MultipleActiveResultSets=true;TrustServerCertificate=True'
& .\.tools\dotnet-tools\dotnet-ef.exe database update `
    --project .\src\EngineeringManager.Infrastructure\EngineeringManager.Infrastructure.csproj `
    --startup-project .\src\EngineeringManager.Web\EngineeringManager.Web.csproj
```

应用运行账号不需要数据库结构修改权限。建议使用单独的部署管理员账号执行迁移，迁移后收回高权限。

### 5.8 IIS 应用池和目录权限

在 IIS 管理器中：

1. 新建应用池 `EngineeringManager`。
2. `.NET CLR 版本` 选择“无托管代码”。
3. 禁用 32 位应用。
4. 将“启动模式”设为 `AlwaysRunning`。
5. 将“加载用户配置文件”设为 `True`。
6. 新建网站，物理路径指向 `D:\EngineeringManager\app`。

授予目录权限：

```powershell
$ErrorActionPreference = 'Stop'
icacls 'D:\EngineeringManager\app' /grant 'IIS AppPool\EngineeringManager:(OI)(CI)(RX)' /T
icacls 'D:\EngineeringManager\app\App_Data' /grant 'IIS AppPool\EngineeringManager:(OI)(CI)(M)' /T
```

### 5.9 SQL Server 最小权限

若使用 IIS 应用池身份连接同机 SQL Server，可由 SQL 管理员执行：

```sql
CREATE LOGIN [IIS APPPOOL\EngineeringManager] FROM WINDOWS;
GO
USE [EngineeringManager];
GO
CREATE USER [IIS APPPOOL\EngineeringManager] FOR LOGIN [IIS APPPOOL\EngineeringManager];
ALTER ROLE [db_datareader] ADD MEMBER [IIS APPPOOL\EngineeringManager];
ALTER ROLE [db_datawriter] ADD MEMBER [IIS APPPOOL\EngineeringManager];
ALTER ROLE [db_backupoperator] ADD MEMBER [IIS APPPOOL\EngineeringManager];
GO
```

应用池身份不应属于 `sysadmin` 或长期属于 `db_owner`。

系统内置数据库备份由 SQL Server 服务执行写盘。必须让 SQL Server 服务账号对备份目录有写权限。SQL Server Express 常见服务账号为 `NT SERVICE\MSSQL$SQLEXPRESS`：

```powershell
$ErrorActionPreference = 'Stop'
icacls 'D:\EngineeringManager\app\App_Data\backups' /grant 'NT SERVICE\MSSQL$SQLEXPRESS:(OI)(CI)(M)' /T
```

实际服务账号可通过 SQL Server Configuration Manager 确认，不要盲目使用示例账号。

### 5.10 IIS 环境变量

在 IIS 的“配置编辑器”中，为站点 `system.webServer/aspNetCore` 的 `environmentVariables` 配置：

```text
ASPNETCORE_ENVIRONMENT=Production
ConnectionStrings__DefaultConnection=<正式连接字符串>
Identity__SeedRoles=true
DevelopmentSampleData__Enabled=false
```

角色模板首次初始化成功后，可以将 `Identity__SeedRoles` 改回 `false`。

当前程序不会自动创建带密码的首位生产管理员。正式投入使用前，必须准备经过审核的一次性管理员创建方式；不得将测试账号或 `local-test-credentials.txt` 用作生产管理员。该项未完成前，只能做部署联调，不能正式交付用户使用。

### 5.11 局域网域名和 HTTPS

推荐使用内网域名，例如：

```text
engineeringmanager.company.lan
```

内网 DNS 将该域名解析到服务器固定 IP。没有内网 DNS 时，可先通过 IP 做短期验证，但正式使用仍应配置域名和证书。

证书可以来自企业内部 CA，或使用受所有客户端信任的证书。自签名证书必须把根证书安装到每台客户端的“受信任的根证书颁发机构”，并确保域名与证书名称一致。

在 IIS 中绑定 `443` 和证书，并将 `80` 重定向到 HTTPS。程序在 Production 环境启用 HSTS 和 HTTPS 重定向，因此证书或绑定错误会导致用户无法正常访问。

### 5.12 Windows 防火墙

仅对专用/域网络开放 HTTPS：

```powershell
$ErrorActionPreference = 'Stop'
New-NetFirewallRule `
    -DisplayName 'EngineeringManager 局域网 HTTPS' `
    -Direction Inbound `
    -Action Allow `
    -Protocol TCP `
    -LocalPort 443 `
    -Profile Private, Domain
```

不要向局域网客户端开放：

- SQL Server `1433`；
- 开发测试端口 `5075`；
- 远程调试端口；
- 不受限制的远程桌面端口。

### 5.13 客户端访问

客户端无需安装 .NET、SQL Server 或项目文件，只需使用 Edge 或 Chrome 访问：

```text
https://engineeringmanager.company.lan
```

手机连接同一局域网 Wi-Fi 后也可使用该地址。

### 5.14 局域网首次验收

```powershell
$ErrorActionPreference = 'Stop'
$baseUrl = 'https://engineeringmanager.company.lan'
Invoke-WebRequest -Uri "$baseUrl/health/live" -UseBasicParsing | Select-Object StatusCode
Invoke-WebRequest -Uri "$baseUrl/health/ready" -UseBasicParsing | Select-Object StatusCode
```

人工检查：

- 首位生产管理员能登录并修改初始密码；
- 不同角色只能看到授权模块和数据；
- 项目、员工、工资、发票、收付款等关键流程可以保存；
- 附件可以上传和下载；
- 系统备份能同时生成数据库备份和附件归档；
- 电脑和手机浏览器均无横向溢出；
- 测试账号、测试库和测试站点没有暴露给正式用户。

## 六、局域网正式系统的日常维护

### 6.1 备份

正式备份至少包含：

- SQL Server 正式数据库完整备份；
- `D:\EngineeringManager\app\App_Data\attachments`；
- 当前发布包和生产配置清单；
- IIS 站点、域名和证书恢复信息；
- 当前 EF Migration 版本。

建议：

- 每日数据库完整备份；
- 每日附件备份或增量归档；
- 每周复制到另一台电脑、NAS 或云存储；
- 每月至少执行一次恢复演练；
- 备份保留周期不少于 30 天，并根据真实业务要求调整。

### 6.2 升级

1. 在开发机完成测试并生成新发布包。
2. 公告维护窗口，停止 IIS 站点。
3. 备份数据库、附件、当前程序目录和配置。
4. 审核并执行 EF Migration。
5. 把新发布包复制到新的 `releases` 目录。
6. 更新 `app` 目录，但保留 `App_Data` 和生产配置。
7. 启动 IIS，检查健康端点、登录和本次变更的关键流程。
8. 观察 IIS、Windows 事件和应用日志后结束维护窗口。

### 6.3 回滚

- 只有程序变化且数据库向后兼容时，才可以仅恢复上一发布包。
- 如果执行了不可逆或不兼容迁移，必须同时恢复迁移前数据库备份和对应程序版本。
- 附件、数据库和程序必须来自同一维护时间点。
- 回滚后检查健康端点、最近业务记录和附件完整性。

## 七、阶段三：部署到云服务器供外网访问

### 7.1 初期推荐架构

业务规模不大时可以先使用一台 Windows 云服务器：

```text
公网用户
   |
正式域名 + HTTPS 443
   |
云安全组 / Windows 防火墙
   |
Windows Server + IIS
   |-- EngineeringManager
   |-- SQL Server
   |-- App_Data
```

数据量或并发增长后，建议把 SQL Server 独立到私网数据库服务器，并把附件迁移到受控共享存储或对象存储。在附件仍使用本机目录时，不要同时运行多个应用实例，否则附件会分散在不同服务器。

### 7.2 云服务器准备

- Windows Server 2022 或更新版本。
- 建议至少 4 核 CPU、8 GB 内存和独立数据盘；正式规格应按真实数据量和并发压测调整。
- 安装 IIS、.NET 10 Hosting Bundle 和 SQL Server，或准备私网 SQL Server。
- 申请正式域名并完成必要的备案、实名认证或合规手续。
- 准备受信任 CA 签发的 HTTPS 证书。
- 准备云备份、监控、告警和秘密管理方案。

### 7.3 云安全组

公网只开放：

- `443/TCP`：所有正式用户；
- `80/TCP`：仅用于跳转 HTTPS 或证书验证；
- `3389/TCP`：只允许固定运维出口 IP 或 VPN，禁止全网开放。

不向公网开放 SQL Server `1433`。应用服务器和数据库服务器之间使用云私网通信。

### 7.4 部署程序和数据库

云端 IIS、应用池、目录权限、环境变量和数据库迁移步骤与局域网正式部署一致，主要区别是：

- 连接字符串改为云端私网 SQL Server 地址；
- 使用云秘密管理或服务器受控环境变量保存连接字符串；
- HTTPS 使用公网域名证书；
- 防火墙同时配置云安全组和 Windows 防火墙；
- 数据库与附件备份复制到另一可用区或另一存储账户。

如果 SQL Server 与 IIS 不在同一台服务器，系统内置 `BACKUP DATABASE` 的目标路径必须是 SQL Server 服务账号可访问的共享路径；否则应关闭应用内人工备份入口，改用 SQL Server 维护计划或云数据库备份服务。

### 7.5 从局域网迁移到云端

正式切换时安排维护窗口：

1. 提前在云端部署并验证空数据库版本。
2. 降低域名 DNS TTL，准备切换。
3. 停止本地 IIS，冻结所有写入。
4. 对本地正式数据库执行最终完整备份并校验。
5. 完整复制附件目录和生产配置清单。
6. 在云端恢复数据库和附件。
7. 核对 `__EFMigrationsHistory`、员工、项目、工资、发票、收付款和附件数量。
8. 启动云端 IIS，使用临时域名或 hosts 记录完成管理员验收。
9. 切换正式 DNS 到云服务器。

## 九、常见问题

### 9.1 `/health/live` 正常但 `/health/ready` 失败

说明应用已启动，但 SQL Server 连接失败。检查：

- SQL Server 服务是否运行；
- 数据库名和实例名是否正确；
- IIS 应用池身份是否已创建 SQL Login/User；
- 连接字符串是否设置到正确站点；
- 云端应用服务器能否通过私网访问数据库。

### 9.2 IIS 返回 500.30 或 500.31

- 检查是否安装匹配的 .NET 10 Hosting Bundle；
- 重启 IIS；
- 查看 Windows 事件查看器和 IIS 日志；
- 检查 `web.config`、环境变量和数据库连接字符串；
- 确认应用池为“无托管代码”且 32 位应用已关闭。

### 9.3 可以打开登录页但附件上传失败

- 检查应用池身份对 `App_Data` 是否有修改权限；
- 检查磁盘剩余空间；
- 检查 IIS 请求大小限制；
- 检查杀毒软件是否阻止发布目录写入。

### 9.4 系统内数据库备份失败

- 确认应用数据库用户属于 `db_backupoperator`；
- 确认 SQL Server 服务账号对 `App_Data/backups` 有写权限；
- 检查备份盘空间和路径是否存在；
- 查看数据库错误和应用日志。

### 9.5 局域网其他电脑无法访问

- 确认客户端和服务器在同一局域网；
- 确认服务器 IP 未变化；
- 检查 Windows 防火墙和 IIS 绑定；
- 使用服务器 IP 测试网络连通，再检查内网 DNS；
- 检查客户端是否信任 HTTPS 证书。

### 9.6 云端域名已解析但无法访问

- 检查 DNS 是否已生效；
- 检查云安全组和 Windows 防火墙的 `443`；
- 检查 IIS HTTPS 绑定和证书；
- 检查公网 IP 是否绑定到正确云服务器；
- 检查应用池和健康端点。
